## MCP ファイル操作の既知の注意（filesystem サーバ共通）

- パス指定は二重バックスラッシュでエスケープする
- 読み取り系がタイムアウトする場合は `read_multiple_files`（単一ファイルでも1要素配列）で代替する（旧構成の `read_file`・`get_file_info` で恒常的に発生。2026-07-04 まで観測）
- 存在確認は list_directory 系で行う（get_file_info のタイムアウト回避）
- list_directory は非再帰。サブディレクトリは個別に呼ぶか `directory_tree` / `search_files` を使う
- 新しいパスへ書き込む前に `create_directory` を呼ぶ
- `write_file` は既存ファイルを警告なしに上書きする。新規作成時は事前に同名ファイルの不在を list_directory 系で確認する
- `edit_file` は直前に対象ファイルを再読込し、oldText は実テキストからの転記とする（記憶やレンダリング表示からの転記は照合失敗の元。波ダッシュ「〜」・不等号「<」（全角「＜」と取り違え）等の類似文字は特に注意。2026-07-04 に「〜」の照合ミスで一括編集が不発。2026-07-05 に不等号「<」を全角と取り違えて照合失敗）
- `edit_file` の複数編集はアトミック: 1件でも照合失敗すると全件不発でファイルは無傷（2026-07-04 観測）。失敗時はファイルを再読込し、正確な文字列で再実行する
- 日本語ファイルを `edit_file` した後は diff を目視確認する（Unicode の稀な破損対策。「冪等性」「趨勢」「圏/圈」「随/隨」等で破損の観測実績）。自分が新規に書き込む日本語にも適用し、異体字に加えて全角/半角の取り違え（全角チルダ「～」・全角数字「１」「３」・全角記号等）も確認する（異体字混入 2026-07-05、全角/半角の自己混入3件 2026-07-05/06 に観測）
- `move_file` が「実行成功・応答エラー（ENOENT）」となることがある。エラー時はリトライ前に list_directory で実状態を確認する（2026-07-04 観測）
- MCP サーバ構成の変更直後は、①`list_allowed_directories` で許可範囲を確認 → ②軽いファイル操作（list_directory 等）で疎通を確認してから本作業に入る（2026-07-05: サーバ統合直後に許可範囲の応答は正常・ファイル操作のみ全滅の状態を観測）
- ファイル操作のタイムアウトが2連続したらリトライを止め、依頼者に MCP サーバ（または AI ツール）の再起動を依頼する（1回4分の待機損失を抑える。2026-07-05: 再起動で回復を確認）
- 日本語ファイルの読み取り表示が文字化けして見えても、即座に内容を修正せず `edit_file` の dry-run で実ファイルの内容を確認する（表示上の問題で実ファイルは正常だったケースがある、2026-07-06: `Hoge` の表示が文字化けして見えたが実ファイルは正常で、誤修正しかけた）

## Cowork（サンドボックス Bash）環境の既知の注意

- マウント先での削除系（`os.remove` / `os.rmdir` / `rm`）は使わず、移動・置換は「内容書換 → `os.rename`（上書きリネーム可）」で行う（2026-07-11: docs 配置換えで削除系のみ EPERM、rename は成功）
- ホスト側ツールで書いた直後のファイルがサンドボックス側の読取で途切れて見えることがある。実破損と判断する前にホスト側 Read で実ファイルを確認する（2026-07-11: 書込直後の index.md が 100 バイトに見えたが実ファイルは正常）
- ホスト側 Edit が「modified since read」エラーになったら、対象ファイルを Read で再読込してから再実行する（フォルダ接続直後に頻発。2026-07-11: 3回観測、いずれも再読込で解消）

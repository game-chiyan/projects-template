# 品質・テスト

> レンズの位置づけ：助言であり決定者ではない。出力は README の「出力の型」に従う。

## 一言ミッション
「出して大丈夫か」を判断し、壊れ方を先に見つける。予防を上流へ寄せる。

## いつ呼ぶ（起動条件）
出荷・リリース判断／テスト設計／異常系・境界の洗い出し／不具合の再発防止。

## 最優先で見る順（既定）
1. 致命的な壊れ方の有無
2. 異常系・境界
3. 再現性
4. 回帰リスク
5. 網羅率（最後）

## 必ず投げる問い
- どこが壊れると致命的か？
- 異常系・境界は洗えているか？
- 再現手順はあるか？
- 直した所の周辺は壊れていないか？
- 出さない判断の基準は？
- これは上流で防げたか？

## よく突く失敗・アンチパターン
ハッピーパスのみ検証／網羅率の自己目的化／再現手順のないバグ報告／リリース後頼み。

## 他レンズとの典型的な対立
- `pm`・`po`（速度）と：出荷 vs 品質確認
- `coder`（実装都合）と：書きやすさ vs 壊し方
- reviewerᴳと協調し、見落とした経路を突く

## 固有の立場（業界の共通見解・たたき台／要調整）
- テストは単体を厚く、結合は程々、E2E/UIは薄く配分する（テストピラミッド）
- テストは自動化してCIで毎回実行し、早い段階で欠陥を捉える（シフトレフト）
- テストは高速・確定的に保ち、フレーキーなテストは修正・排除する
- テストコードも本番同等の品質で書き、保守対象として扱う

> 複数の出典に共通する既定をたたき台として記入。実運用で自分・組織の方針に上書きする（出典は SOURCES.md）。

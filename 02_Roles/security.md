# セキュリティ

> レンズの位置づけ：助言であり決定者ではない。出力は README の「出力の型」に従う。

## 一言ミッション
悪用・漏洩・侵害の面から最悪を想定し、守りの抜けを塞ぐ。

## いつ呼ぶ（起動条件）
本番・データ・認証・外部公開・権限に触れる判断／個人情報の扱い／依存の追加。

## 最優先で見る順（既定）
1. 機密・個人情報の露出
2. 認証・認可
3. 入力と信頼境界
4. 秘密情報の管理
5. 利便性（最後）

## 必ず投げる問い
- 攻撃者なら何を狙う？
- このデータが漏れたら何が起きる？
- 誰がアクセスできるべきで、実際そうなっているか？
- 入力を信頼していないか？
- 秘密情報はどこに置かれている？
- 最小権限になっているか？

## よく突く失敗・アンチパターン
後付けのセキュリティ／秘密のハードコード／過剰な権限／入力の無検証／「社内だから安全」。

## 他レンズとの典型的な対立
- `po`・`pm`（速度）と：守り vs リリース
- `ux-designer`（利便性）と：安全 vs 手数
- `reviewer` と協調し、攻撃面を突く

## 固有の立場（業界の共通見解・たたき台／要調整）
- 設計初期からセキュリティを組み込み、既定値を安全側にする（セキュア・バイ・デフォルト）
- 最小権限を既定にし、アクセス制御はサーバ側で最小限に絞る
- 秘密情報はコードに書かず、分離して集中管理しローテーションする
- 全入力を検証し、出力は文脈に応じてエスケープしてインジェクションを防ぐ

> 複数の出典に共通する既定をたたき台として記入。実運用で自分・組織の方針に上書きする（出典は SOURCES.md）。

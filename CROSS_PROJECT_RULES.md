# CROSS_PROJECT_RULES.md（プロジェクト横断ルール）

- 対象: `~\Projects` 配下の全プロジェクト
- 位置づけ: 本書 = 全プロジェクト共通ルール。AIツール別の入口は `AGENTS.md` / `CLAUDE.md` に置く
- プロジェクト固有のルールは `~\Projects\<プロジェクト>\PROJECT_RULES.md` に置く
- 優先順位: 記載が矛盾する場合はプロジェクト側 `PROJECT_RULES.md` が優先する（具体 > 抽象）

## CROSS_PROJECT_RULES.md を育てるルール（メタルール）

1. CR-001: 追記トリガー: ①依頼者から指摘を受けた ②同種の失敗が2回目に達した（1回目は失敗記録台帳に `様子見` で記録する）③有効な回避策・手順を発見した
2. CR-002: 分析様式: 「事象 → 原因 → 仕組み化」の3点で1件ずつ分析する。ルール本文は短文の行動形（「〜する」）で書き、末尾に根拠を括弧書き（日付・事象）で残す
3. CR-003: 禁止形より代替形: 「Xしない」ではなく「Yする（Xの代わりに）」の形で書く。行動が指定されている方が失敗しにくい
4. CR-004: 配置判定: 「他プロジェクトでも再現し得るか」で判定する。Yes → 本書、No → プロジェクト側。AIツール固有なら `AGENTS.md` または `CLAUDE.md`（状況限定の詳細は `~\Projects\04_Rules_Reference\`）に置く
5. CR-005: 更新フロー: 追記・変更・削除はすべて提案 → 依頼者承認後に反映する
6. CR-006: 振り返り: セッション終了時に当該セッションの失敗・指摘を振り返り、台帳への記録と追記候補を提示する
7. CR-007: 局所棚卸し: ルール追記の承認時に、同セクションの既存ルールも見直して改訂・削除候補を提示する（2026-07-12 指示）
8. CR-008: 全体棚卸し: 前回棚卸し時点から本書が30行増えたら全体棚卸しを提案する（基準: 2026-07-14 時点 106行。棚卸し実施時に基準を更新する）
9. CR-009: 有効性判定: 棚卸しでは各ルールを「再発あり → 表現・仕組みを改訂」「再発なし → 維持（重要なものは入口ファイルに要点を残して強化）」「前提消滅・プロジェクト固有と判明 → 削除または `PROJECT_RULES.md` へ格下げ」に分類し、判定案を提示して依頼者が裁定する（2026-07-12 指示）
10. CR-010: 肥大化抑制: 1ルール1〜2行とする。背景説明が長くなる場合は詳細を別文書に置き、本書からはリンクで参照する
11. CR-011: パス表記: `CROSS_PROJECT_RULES.md` や `PROJECT_RULES.md` にはフルパスを記載せず、`Projects` より上位ディレクトリは `~\` として記載する（2026-07-09 指示）
12. CR-012: 固有名称の匿名化: 本書へメタルールによる自己成長のためのルールを追記する際は、各プロジェクト内の固有名称を記載せず、例示に文字列が必要な場合は `Hoge` のように隠す（2026-07-09 指示）
13. CR-013: ID採番: ルールはファイル別接頭辞（CR / CL / AG / MF / CS / OR / PJ）＋3桁の通し番号で識別する。IDは行動ルールに付与し、事実・環境情報には付与しない。新規追記は当該ファイルの既存最大番号+1とし、欠番は詰めず、削除した番号は再利用しない。台帳・handoverᴳ からルールを参照する際はIDを用いる（2026-07-14 導入）

## 失敗記録台帳

- CR-014: 失敗・指摘の1回目は `~\Projects\<プロジェクト>\incident-log.md` に記録する（handoverᴳ ディレクトリと同階層、リポジトリ外。2026-07-12 指示）
- CR-015: プロジェクトに属さない作業（横断ルール整備・`00_Template` 保守等）は `~\Projects\incident-log.md` に記録する
- CR-016: 記載項目: 日付 / 事象 / 原因 / 暫定対処 / 状態 / 関連ルール。状態は `様子見`（1回目）/ `昇格済`（ルール化）/ `再発`（ルール化後も発生）の3値。関連ルール欄はルールID（例: CR-024）で記載する
- CR-017: ルール化済みの失敗が再発したら状態を `再発` にし、関連ルール欄に対象ルールを記載する
- CR-018: プロジェクト側の台帳は初回の記録時に `~\Projects\incident-log.md` を雛形として作成する（空の台帳を事前に作らない）

## 作業ルール

- CR-019: 何か作業する際は必ず確認を取ってから行う（文脈的にOKを出している場合と、CR-063 の事後報告対象操作を除く）
- CR-020: リファクタリング時は気になった点をすべて指摘する
- CR-021: 変数や関数などをリネームする際は使用箇所を確認し、すべて対応する
- CR-022: 変数名や関数名は正式名称を用いる（略称の代わりに）
- CR-023: 情報は記録先を役割で分ける: コードに How（実現方法）、テストコードに What（仕様）、コミットログに Why（変更理由）、コードコメントに Why not（あえて選ばなかった代替案とその理由）を書く。Why not を明示し「知らずにやった」との誤読を防ぐ（t_wada 氏提唱。2026-07-12 指示）
- CR-024: ドキュメント内の空白は半角スペースを用いる（全角スペースは使わない。2026-07-13 指示）
- CR-063: 次の可逆・記録系操作は事前確認なしで実施し、実施後に報告する: `worklog.md` への追記 / `resume.md` の最新化 / 承認済みタスクᴳ範囲内の handoverᴳ ファイル作成。不可逆操作・対外反映・ルール変更・リポジトリ内コードの変更は従来通り事前承認とする（承認を本質的判断に集中させる。判断軸「可逆な決定は速く」に整合。2026-07-14 導入）

## AI開発ルール

- CR-025: AI生成コードは対応するテストと同梱でのみ受け入れる（2026-07-11 プロジェクト側 AI開発規約から昇格）
- CR-026: マージ前に人間が全 diff を読む。理解できない箇所は AI に説明させ、腹落ちしなければマージしない（同上）
- CR-027: 新規依存パッケージは AI が理由を提示し、人間の承認後に追加する（同上）
- CR-028: AI が主に書いたコミットはメッセージ末尾に `[ai]` を付ける（品質傾向の事後分析用。同上）
- CR-029: シークレット全般とマスク前の本番ログを AI に渡さない。渡してよいのはリポジトリ内のコード・設計書・録画フィクスチャ・公開仕様・匿名化したエラーログ（同上）
- CR-069: 事実主張のうちコードベース外の断定（ライブラリ / API / バージョン / 外部仕様）と下流影響の大きい重要判断（技術選定・影響範囲）には根拠（`file:line` / 実行コマンド出力 / 公式ドキュメント URL）を併記し、確認できないものは `要検証` として検証待ちキューᴳ（CR-066）へ回す。内部コードの事実は構文確認・テスト（CR-025 / CR-065）と別モデル監査（CR-034）に委ね二重化しない。適用箇所（フェーズᴳ 2/3/4 のステップᴳ・完了条件）は session-phases.md に従う（AIDD のハルシネーション対策として根拠確認を挟む。2026-07-16 導入）

## 判断レンズ運用（02_Roles）

- CR-030: 重要な判断（設計・技術選定・優先度・リリース可否など）では、`~\Projects\02_Roles\` の関連レンズ2〜3個と `reviewer`（批判者）を併用し、観点をトレードオフで出してから決める（一つの声に流されないため。2026-07-13 指示）
- CR-031: レンズの意見が割れたら `~\Projects\02_Roles\_judgment-axis.md`（判断軸）に照らして決める。決まらなければ同ファイル末尾の手続きに従う
- CR-032: レンズは助言であり決定者ではない。最終判断は依頼者が下す
- CR-033: 各レンズの「固有の立場」は業界標準のたたき台。運用で方針が定まったら該当レンズに追記し、末尾の最終確認日を更新する
- CR-034: Cowork では `reviewer` を生成と別系統のモデルで起動して監査する。手順は `~\Projects\04_Rules_Reference\orchestration.md`「監査フロー」を参照する（同一モデルの相関した盲点を避ける。2026-07-14 指示）
- CR-035: レンズ一覧・起動条件・出力の型・更新規律の詳細は `~\Projects\02_Roles\README.md` を参照する

## ドキュメント参照範囲の運用

- CR-037: リポジトリ内ドキュメントは同一リポジトリ内で完結させ、上位階層（プロジェクト階層・横断階層）のファイルへ参照を残さない。上位階層の情報が必要なら要点をリポジトリ内へ転記する（単体 clone・CI 時に参照先が壊れるため。2026-07-13 判明）
- CR-038: プロジェクト階層のドキュメント（`PROJECT_RULES.md`・`handover\`・`temp\`等）からリポジトリ内ドキュメントへの参照は可とする（一方向。逆方向は上記により不可）
- CR-039: `~\Projects` 直下の横断ドキュメントと個別プロジェクトディレクトリは相互参照しない。`NN_`接頭辞ディレクトリ（プロジェクト横断の共有リソース）への参照は例外的に可とする
- CR-064: 数値・基準等の正ᴳは1文書にのみ記載し、他文書からは参照で引く（正の一元化ᴳ。重複記載をしない。2026-07-14 Auto_Trader の固有ルールから昇格）
- CR-067: 用語を定義するときは、定義の正を `~\Projects\04_Rules_Reference\glossary.md` に集約し、本文で定義済みの意味で使う箇所には上付き ᴳ（U+1D33）を付す。一般的な意味の同語には付けない。記法・付与運用の詳細は glossary.md に従い、ᴳ 付き語の実在は 03_Checks の glossary-terms で検査する（定義の意味かは機械判定できないため付与は書き手が行う。2026-07-16 導入）

## セッションルール

- CR-047: セッションはフェーズᴳの節目で区切る。1 セッションで連続する複数フェーズᴳを進め、handoverᴳ を書く節目（CR-058）かつコンテキストが肥大化してきた時点で分割する。1 タスクᴳを必ず 1 セッションに収める必要はない（2026-07-15 改訂。旧: 1 タスクᴳ 1 セッション運用）
- CR-048: 分割時は resume.md と直近 handoverᴳ を精査し、再開に必要な情報が揃っていることを確認してから切り替えを提案する
- CR-049: セッションは 開始手続きᴳ → タスク定義 → ブランチ名・チャット名提案 → 要件定義 → 基本設計 → 詳細設計 → テスト作成 → 実装 → リファクタ → ドキュメント整備 の順で進行する。各フェーズᴳのインプット・ステップᴳ・アウトプット・完了条件と成果物テンプレートは `~\Projects\04_Rules_Reference\session-phases.md` に従う（アウトプットを次フェーズのインプットにそろえる。2026-07-15 詳細化）
- CR-050: タスクᴳが仕様書作成の場合は、実装、テスト作成、リファクタは不要
- CR-051: 途中の確認は随時挟む
- CR-052: 各プロジェクトの handoverᴳ ディレクトリに継続作業ログを2層で持つ（`00_Template` を除く）: `worklog.md`（追記専用の作業履歴）と `resume.md`（現在の再開地点を1枚に上書き保存）
- CR-068: プロジェクトに属さない横断作業（`~\Projects` 直下・`NN_` 配下のルール／ドキュメント整備、`00_Template` 保守）では `worklog.md`・`resume.md`・節目 handoverᴳ を作らず、記録は `~\Projects\incident-log.md` に集約する（CR-015）。run-checks（CR-062）は対象範囲に従来どおり実施する。CR-052 の2層ログはプロジェクト作業を対象とする（横断作業には専用 handoverᴳ ディレクトリがなく2層ログの前提が成立しないため。2026-07-16 指示）
- CR-054: チェックポイントᴳごとに `worklog.md` へ末尾追記し、`resume.md` を最新化する。長時間・不可逆な操作の前と、中断・ツール切替・セッション終了の前にも必ず更新する（作業ごとに履歴を残し任意のツール・セッションへ即引き継ぐため。従来は終了時のみ作成で中断時に引き継げなかった。2026-07-14 指示）
- CR-055: `worklog.md` の各エントリには 日時 / 実行ツール・セッション識別 / 実施内容 / 変更ファイル / 決定と理由 / 検証（CR-065 の DoDᴳ 消込形式） / 次の一手 を記す（どのツールが何をしたか追跡するため）
- CR-056: `resume.md` には 更新時刻 / 実行ツール・セッション / 対象ブランチ / 現在タスクとゴール / 直近完了ステップᴳ / 次の一手 / 進行中の途中状態 / 未解決・ブロッカー / 検証待ちキューᴳ（CR-066） / 参照先 を短く記す
- CR-057: セッション開始時はプロジェクト側 `PROJECT_RULES.md` → `resume.md`（あれば）→ 必要に応じ `worklog.md` 末尾 の順で読み、再開地点を把握してから着手する。再読込は resume.md のみを既定とし、handoverᴳ・worklogᴳ はブロッカー解消や詳細確認が必要なときだけ引く（分割時の初期コストを抑えるため）
- CR-058: 節目（セッション単位のタスク・機能の完了、区切りの良い停止）で `handover-YYYY-MM-DD[-N].md` を handoverᴳ ディレクトリ（`~\Projects\<プロジェクト>\handover\`）に作成し、当該区間の worklogᴳ を要約・保全する。完了事項・次タスクᴳの作業順・未解決事項・運用ノートを記載する。毎回の停止では必須としない
- CR-059: 肥大化時は当該区間を節目 handoverᴳ へ要約し、`worklog.md` を `worklog-YYYY-MM.md` へ退避して本体を軽く保つ。`resume.md` は常に短く保つ
- CR-060: セッション開始時に、タスク内容が分かるチャット名（例: `Hogeの実装`）を提示し、依頼者がチャット名に設定する（チャット名変更は依頼者操作。2026-07-05 指示）
- CR-062: セッション終了時に `node ~\Projects\03_Checks\run-checks.js <対象>` を実行し、違反を修正または依頼者へ報告する。依頼者はコミット前にローカルで実行し、その結果を正とする（サンドボックス実行は読取途切れ CS-002 の誤検出があり得るため。2026-07-14 導入）

## 完了の定義（DoD）

- CR-065: タスク種別ごとの必須検証を下表の通りとし、worklogᴳ の「検証」欄は該当項目の消込形式（実施済みは結果、未実施は理由と代替）で記載する（検証内容の書き手依存をなくす。2026-07-14 導入）
- CR-066: 環境制約で実施できない検証は `resume.md` の「検証待ちキューᴳ」欄へ記録し、解消時に消し込む。キューに残がある間はタスクᴳを完了扱いにしない（サンドボックスで lint/test が完走しない恒常事象に対応。2026-07-14 導入）

| タスク種別 | 必須検証 |
| --- | --- |
| 実装・コード変更 | 構文確認（`node --check` 等）/ lint / テスト実行（テスト同梱 = CR-025）。実行不可なら検証待ちキューᴳへ |
| ドキュメント作成・編集 | 文字点検（03_Checks doc-chars）/ 参照リンク・パスの整合 / 編集後の差分目視（CL-003 / AG-003） |
| ルール変更 | ID採番の整合（CR-013）/ 参照元（台帳・入口・README）の更新確認 / 03_Checks 実行（CR-062） |
| 仕様書・設計資料 | 設計の正ᴳとの整合 / 数値・基準の一元化確認（CR-064）/ 依頼者承認 |

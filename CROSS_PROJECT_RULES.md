# CROSS_PROJECT_RULES.md（プロジェクト横断ルール）

- 対象: `~\Projects` 配下の全プロジェクト
- 位置づけ: 本書 = 全プロジェクト共通ルール。AIツール別の入口は `AGENTS.md` / `CLAUDE.md` に置く
- プロジェクト固有のルールは `~\Projects\<プロジェクト>\PROJECT_RULES.md` に置く
- 優先順位: 記載が矛盾する場合はプロジェクト側 `PROJECT_RULES.md` が優先する（具体 > 抽象）

## CROSS_PROJECT_RULES.md を育てるルール（メタルール）

1. 追記トリガー: ①依頼者から指摘を受けた ②同種の失敗が2回目に達した（1回目は失敗記録台帳に `様子見` で記録する）③有効な回避策・手順を発見した
2. 分析様式: 「事象 → 原因 → 仕組み化」の3点で1件ずつ分析する。ルール本文は短文の行動形（「〜する」）で書き、末尾に根拠を括弧書き（日付・事象）で残す
3. 禁止形より代替形: 「Xしない」ではなく「Yする（Xの代わりに）」の形で書く。行動が指定されている方が失敗しにくい
4. 配置判定: 「他プロジェクトでも再現し得るか」で判定する。Yes → 本書、No → プロジェクト側。AIツール固有なら `AGENTS.md` または `CLAUDE.md` に置く
5. 更新フロー: 追記・変更・削除はすべて提案 → 依頼者承認後に反映する
6. 振り返り: セッション終了時に当該セッションの失敗・指摘を振り返り、台帳への記録と追記候補を提示する
7. 局所棚卸し: ルール追記の承認時に、同セクションの既存ルールも見直して改訂・削除候補を提示する（2026-07-12 指示）
8. 全体棚卸し: 前回棚卸し時点から本書が30行増えたら全体棚卸しを提案する（基準: 2026-07-12 時点 85行。棚卸し実施時に基準を更新する）
9. 有効性判定: 棚卸しでは各ルールを「再発あり → 表現・仕組みを改訂」「再発なし → 維持（重要なものは入口ファイルに要点を残して強化）」「前提消滅・プロジェクト固有と判明 → 削除または `PROJECT_RULES.md` へ格下げ」に分類し、判定案を提示して依頼者が裁定する（2026-07-12 指示）
10. 肥大化抑制: 1ルール1〜2行とする。背景説明が長くなる場合は詳細を別文書に置き、本書からはリンクで参照する
11. パス表記: `CROSS_PROJECT_RULES.md` や `PROJECT_RULES.md` にはフルパスを記載せず、`Projects` より上位ディレクトリは `~\` として記載する（2026-07-09 指示）
12. 固有名称の匿名化: 本書へメタルールによる自己成長のためのルールを追記する際は、各プロジェクト内の固有名称を記載せず、例示に文字列が必要な場合は `Hoge` のように隠す（2026-07-09 指示）

## 失敗記録台帳

- 失敗・指摘の1回目は `~\Projects\<プロジェクト>\incident-log.md` に記録する（handover ディレクトリと同階層、リポジトリ外。2026-07-12 指示）
- プロジェクトに属さない作業（横断ルール整備・`00_Template` 保守等）は `~\Projects\incident-log.md` に記録する
- 記載項目: 日付 / 事象 / 原因 / 暫定対処 / 状態 / 関連ルール。状態は `様子見`（1回目）/ `昇格済`（ルール化）/ `再発`（ルール化後も発生）の3値
- ルール化済みの失敗が再発したら状態を `再発` にし、関連ルール欄に対象ルールを記載する
- プロジェクト側の台帳は初回の記録時に `~\Projects\incident-log.md` を雛形として作成する（空の台帳を事前に作らない）

## 作業ルール

- 何か作業する際は必ず確認を取ってから行う（文脈的にOKを出している場合はその限りではない）
- リファクタリング時は気になった点をすべて指摘する
- 変数や関数などをリネームする際は使用箇所を確認し、すべて対応する
- 変数名や関数名には略称を用いない
- 情報は記録先を役割で分ける: コードに How（実現方法）、テストコードに What（仕様）、コミットログに Why（変更理由）、コードコメントに Why not（あえて選ばなかった代替案とその理由）を書く。Why not を明示し「知らずにやった」との誤読を防ぐ（t_wada 氏提唱。2026-07-12 指示）
- ドキュメント内の空白は半角スペースを用いる（全角スペースは使わない。2026-07-13 指示）

## AI開発ルール

- AI生成コードは対応するテストと同梱でのみ受け入れる（2026-07-11 プロジェクト側 AI開発規約から昇格）
- マージ前に人間が全 diff を読む。理解できない箇所は AI に説明させ、腹落ちしなければマージしない（同上）
- 新規依存パッケージは AI が理由を提示し、人間の承認後に追加する（同上）
- AI が主に書いたコミットはメッセージ末尾に `[ai]` を付ける（品質傾向の事後分析用。同上）
- シークレット全般とマスク前の本番ログを AI に渡さない。渡してよいのはリポジトリ内のコード・設計書・録画フィクスチャ・公開仕様・匿名化したエラーログ（同上）

## 判断レンズ運用（02_Roles）

- 重要な判断（設計・技術選定・優先度・リリース可否など）では、`~\Projects\02_Roles\` の関連レンズ2〜3個と `reviewer`（批判者）を併用し、観点をトレードオフで出してから決める（一つの声に流されないため。2026-07-13 指示）
- レンズの意見が割れたら `~\Projects\02_Roles\_judgment-axis.md`（判断軸）に照らして決める。決まらなければ同ファイル末尾の手続きに従う
- レンズは助言であり決定者ではない。最終判断は依頼者が下す
- 各レンズの「固有の立場」は業界標準のたたき台。運用で方針が定まったら該当レンズに追記し、末尾の最終確認日を更新する
- レンズ一覧・起動条件・出力の型・更新規律の詳細は `~\Projects\02_Roles\README.md` を参照する

## ディレクトリ運用（Projects 直下）

- プロジェクト進行用の共通ディレクトリは `NN_`（数字2桁＋`_`）を接頭辞にする。`.gitignore` が `/[0-9][0-9]_*/` で追跡許可するため、追加時に `.gitignore` を編集しない（2026-07-13 指示）

## ドキュメント参照範囲の運用

- リポジトリ内ドキュメントは同一リポジトリ内で完結させ、上位階層（プロジェクト階層・横断階層）のファイルへのリンク・参照を残さない（リポジトリ単体でclone・CI実行される場合に参照先が存在せず壊れるため。上位階層の情報が必要な場合は要点をリポジトリ内へ転記する。2026-07-13: リポジトリ内ドキュメントからリポジトリ外のプロジェクト管理ドキュメントへの相対リンクが、参照先の実在位置と食い違い機能しない状態で記載されていたことが判明）
- プロジェクト階層のドキュメント（`PROJECT_RULES.md`・`handover\`・`temp\`等）からリポジトリ内ドキュメントへの参照は可とする（一方向。逆方向は上記により不可）
- `~\Projects` 直下の横断ドキュメントと個別プロジェクトディレクトリは相互参照しない。`NN_`接頭辞ディレクトリ（プロジェクト横断の共有リソース）への参照は例外的に可とする

## テンプレート運用

- `~\Projects\00_Template\` はテンプレート管理用コンテナ、`~\Projects\00_Template\repository-template\` は各リポジトリへ複製する雛形として扱い、設計資料・GitHub Pages公開環境・リポジトリ運用に必須なファイルだけを置く（2026-07-10 指示）
- 調査用ファイル、一時メモ、サンプルの追加資料、生成物はコミット対象に含めず、`00_Template` の保守作業では引継ぎファイルを作成しない
- ファイルや依存関係を追加する前に、テンプレート利用時に常時必要かを確認し、不要なものは作業完了時に削除するか、ローカル実行に必要な場合は `.gitignore` に記載してコミット対象から除外する

## Docusaurus ドキュメント運用

- ローカル横断閲覧用の Docusaurus + Mermaid 実行環境は `~\Projects\01_Docs_Portal\` に集約する
- GitHub Pagesで各リポジトリの資料を個別公開する場合は、リポジトリ直下の `website\` に最小のDocusaurus設定・依存関係・ロックファイルを置き、`.github\workflows\` の専用workflowでビルド・デプロイする（2026-07-10 指示）
- `website\node_modules\`、`website\.docusaurus\`、`website\build\` は `.gitignore` に記載し、`website\package-lock.json` はCIの再現性確保のためコミット対象に含める
- 各リポジトリの設計資料は `~\Projects\<プロジェクト>\<リポジトリ>\docs\` に置き、ポータルトップ用の `docs\index.md` を必ず作成する（2026-07-10 指示）
- `docs\index.md` にはプロジェクト概要と主要資料へのリンクを記載し、中央ポータルからプロジェクトトップとして参照できる状態にする
- `docs\` のディレクトリ構成は自動生成サイドバーの構成と一致させ、`architecture\`、`design\`、`adr\`、`operations\`、`assets\` など必要な分類だけを作成する
- カテゴリの表示名と順序は `_category_.json`、文書の表示名と順序は Front Matter の `title` と `sidebar_position` で指定する
- 文書間リンクは `.md` を含む相対パスで記載し、画像や配布ファイルは `docs\assets\` に置く。`docs\` 外のファイルはリポジトリのWeb URLで参照する
- 設計資料は Markdown で記述し、構造・処理・シーケンスなどの図は Mermaid を使用する
- Docusaurusのドキュメントバージョニングは必要性が明確になった場合だけ導入し、通常はGit履歴で変更を管理する
- 既存リポジトリの非準拠状態は当面許容し、新規作成または今後のドキュメント整理時に本ルールへ段階的に合わせる（2026-07-10 指示）

## セッションルール

- 基本的に1タスクにつき1セッションで運用する
- セッションが肥大化してきた場合、引継ぎ内容を精査した上でセッションの切り替えを提案する
- セッションは対象タスクと現状の把握 → ブランチ名の提案 → 仕様相談・要件定義 → 設計 → 実装 → テストの作成 → リファクタ → ドキュメント整備の流れで進行する
- タスクが仕様書作成の場合は、実装、テスト作成、リファクタは不要
- 途中の確認は随時挟む
- `00_Template` の保守を除き、セッション終了時は引継ぎファイル `handover-YYYY-MM-DD[-N].md` をプロジェクトの handover ディレクトリ（`~\Projects\<プロジェクト>\handover\`）に作成する。完了事項・次タスクの作業順・未解決事項・運用ノートを記載する
- セッション開始時はプロジェクト側 `PROJECT_RULES.md` と、存在する場合は最新の引継ぎファイルを読んでから着手する
- セッション開始時に、タスク内容が分かるチャット名（例: `Hogeの実装`）を提示し、依頼者がチャット名に設定する（チャット名変更は依頼者操作。2026-07-05 指示）

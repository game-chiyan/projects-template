# CROSS_PROJECT_RULES.md（プロジェクト横断ルール）

- 対象: `~\Projects` 配下の全プロジェクト
- 位置づけ: 本書 = 全プロジェクト共通ルール。AIツール別の入口は `AGENTS.md` / `CLAUDE.md` に置く
- プロジェクト固有のルールは `~\Projects\<プロジェクト>\PROJECT_RULES.md` に置く
- 優先順位: 記載が矛盾する場合はプロジェクト側 `PROJECT_RULES.md` が優先する（具体 > 抽象）

## CROSS_PROJECT_RULES.md を育てるルール（メタルール）

1. 追記トリガー: ①依頼者から指摘を受けた ②同種の失敗が2回目に達した（1回目はセッション引継ぎに記録して様子見）③有効な回避策・手順を発見した
2. 分析様式: 「事象 → 原因 → 仕組み化」の3点で1件ずつ分析する。ルール本文は短文の行動形（「〜する」）で書き、末尾に根拠を括弧書き（日付・事象）で残す
3. 禁止形より代替形: 「Xしない」ではなく「Yする（Xの代わりに）」の形で書く。行動が指定されている方が失敗しにくい
4. 配置判定: 「他プロジェクトでも再現し得るか」で判定する。Yes → 本書、No → プロジェクト側。AIツール固有なら `AGENTS.md` または `CLAUDE.md` に置く
5. 更新フロー: 追記・変更・削除はすべて提案 → 依頼者承認後に反映する
6. 棚卸し: セッション引継ぎ作成時に当該セッションの失敗・指摘を振り返り、追記候補を提示する。大きな節目で陳腐化・重複したルールの削除を提案する
7. 肥大化抑制: 1ルール1〜2行とする。背景説明が長くなる場合は詳細を別文書に置き、本書からはリンクで参照する
8. パス表記: `CROSS_PROJECT_RULES.md` や `PROJECT_RULES.md` にはフルパスを記載せず、`Projects` より上位ディレクトリは `~\` として記載する（2026-07-09 指示）
9. 固有名称の匿名化: 本書へメタルールによる自己成長のためのルールを追記する際は、各プロジェクト内の固有名称を記載せず、例示に文字列が必要な場合は `Hoge` のように隠す（2026-07-09 指示）

## 作業ルール

- 何か作業する際は必ず確認を取ってから行う（文脈的にOKを出している場合はその限りではない）
- リファクタリング時は気になった点をすべて指摘する
- 変数や関数などをリネームする際は使用箇所を確認し、すべて対応する
- 変数名や関数名には略称を用いない

## セッションルール

- 基本的に1タスクにつき1セッションで運用する
- セッションが肥大化してきた場合、引継ぎ内容を精査した上でセッションの切り替えを提案する
- セッションは対象タスクと現状の把握 → ブランチ名の提案 → 仕様相談 → 実装 → テストの作成 → リファクタ → ドキュメント整備の流れで進行する
- 仕様書作成フェーズではブランチ名の提案と実装、テスト作成、リファクタは不要
- 途中の確認は随時挟む
- セッション終了時は引継ぎファイル `handover-YYYY-MM-DD[-N].md` をプロジェクトの handover ディレクトリ（`~\Projects\<プロジェクト>\handover\`）に作成する。完了事項・次タスクの作業順・未解決事項・運用ノートを記載する
- セッション開始時は最新の引継ぎファイルとプロジェクト側 `PROJECT_RULES.md` を読んでから着手する
- セッション開始時に、タスク内容が分かるチャット名（例: `Hogeの実装`）を提示し、依頼者がチャット名に設定する（チャット名変更は依頼者操作。2026-07-05 指示）

## 禁止事項

- Git 操作はしない（コミット・プッシュ等は依頼者が実施する）

## 開発環境

- Windows
- コマンドプロンプト
- Git

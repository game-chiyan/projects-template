# CROSS_PROJECT_RULES.md（プロジェクト横断ルール）

- 対象: `~\Projects` 配下の全プロジェクト
- 位置づけ: 本書 = 全プロジェクト共通ルール。AIツール別の入口は `AGENTS.md` / `CLAUDE.md` に置く
- プロジェクト固有のルールは `~\Projects\<プロジェクト>\PROJECT_RULES.md` に置く
- 文書の役割・正ᴳ・参照トリガーは `~\Projects\04_Rules_Reference\README.md` を正ᴳとする
- 優先順位: 依頼者の明示指示 → 対象プロジェクトの `PROJECT_RULES.md` → 適用中の状況別ツールルール → ツール入口 → 本書。同一階層では具体的な規則を優先する

## CROSS_PROJECT_RULES.md を育てるルール（メタルール）

1. CR-001: 追記トリガー: ①依頼者から指摘を受けた ②同種の失敗が2回目に達した（1回目は失敗記録台帳に `様子見` で記録する）③有効な回避策・手順を発見した
2. CR-002: 分析様式: 「事象 → 原因 → 仕組み化」の3点で分析する。ルール本文は短い行動形にし、日付・事例・経緯は失敗記録台帳からルールIDで参照する
3. CR-003: 禁止形より代替形: 「Xしない」ではなく「Yする（Xの代わりに）」の形で書く。行動が指定されている方が失敗しにくい
4. CR-004: 配置判定: 「他プロジェクトでも再現し得るか」で判定する。Yes → 本書、No → プロジェクト側。AIツール固有なら `AGENTS.md` または `CLAUDE.md`、状況限定の詳細は `~\Projects\04_Rules_Reference\` に置く。文書種別ごとの責務は CR-072 に従う
5. CR-005: 更新フロー: 追記・変更・削除はすべて提案 → 依頼者承認後に反映する
6. CR-006: 振り返り: セッション終了時に当該セッションの失敗・指摘を振り返り、台帳への記録と追記候補を提示する
7. CR-007: 局所棚卸し: ルール追記の承認時に、同セクションの既存ルールも見直して改訂・削除候補を提示する（2026-07-12 指示）
8. CR-008: 全体棚卸し: 前回棚卸し時点から本書が1,500文字または10ルール増えたら全体棚卸しを提案し、棚卸し後の文字数・ルール数を新しい基準として記録する（基準: 2026-07-18 9,222文字・72ルール）
9. CR-009: 有効性判定: 棚卸しでは各ルールを「再発あり → 表現・仕組みを改訂」「再発なし → 維持（重要なものは入口ファイルに適用条件とID参照を残す）」「前提消滅・プロジェクト固有と判明 → 削除または `PROJECT_RULES.md` へ格下げ」に分類し、判定案を提示して依頼者が裁定する
10. CR-010: 肥大化抑制: ルール本文は行動と適用条件を1〜2行で記載する。事例・履歴・詳細手順は台帳または `~\Projects\04_Rules_Reference\` に置き、本書から参照する
11. CR-011: パス表記: `CROSS_PROJECT_RULES.md` や `PROJECT_RULES.md` にはフルパスを記載せず、`Projects` より上位ディレクトリは `~\` として記載する（2026-07-09 指示）
12. CR-012: 固有名称の匿名化: 横断共有文書には各プロジェクト固有の名称を記載せず、例示は `Hoge` や `<プロジェクト>` のようなプレースホルダーにする
13. CR-013: ID採番: ルールはファイル別接頭辞（CR / CL / AG / MF / CS / OR / PJ）＋3桁の通し番号で識別する。IDは行動ルールに付与し、事実・環境情報には付与しない。新規追記は当該ファイルの既存最大番号+1とし、欠番は詰めず、削除した番号は再利用しない。台帳・handoverᴳ からルールを参照する際はIDを用いる。IDの大小は優先順位・適用順を示さない（2026-07-14 導入）
14. CR-072: 文書責務: 本書は適用条件と必須行動、`04_Rules_Reference` は参照先として指定された規範詳細、`glossary.md` は用語の意味、テンプレートは必須成果物の記入構造、`03_Checks` は機械検証可能な一部の実装を担う。詳細・テンプレート・検査は上位ルールを弱めたり新しい義務を単独で作ったりせず、正ᴳと参照トリガーは `~\Projects\04_Rules_Reference\README.md` に従う

## 失敗記録台帳

- CR-014: 失敗・指摘の1回目は `~\Projects\<プロジェクト>\incident-log.md` に記録する（`handover\` ディレクトリと同階層、リポジトリ外。2026-07-12 指示）
- CR-015: プロジェクトに属さない作業で発生した失敗・指摘は `~\Projects\incident-log.md` に記録する
- CR-016: 記載項目: 日付 / 事象 / 原因 / 暫定対処 / 状態 / 関連ルール。状態は `様子見`（1回目）/ `昇格済`（ルール化）/ `再発`（ルール化後も発生）の3値。関連ルール欄はルールID（例: CR-024）で記載する
- CR-017: ルール化済みの失敗が再発したら状態を `再発` にし、関連ルール欄に対象ルールを記載する
- CR-018: プロジェクト側の台帳は初回の記録時に `~\Projects\04_Rules_Reference\incident-log-template.md` から作成する（既存事故を含む横断台帳は複製しない）

## 作業ルール

- CR-019: タスクᴳ開始時に目的・対象・対象外・制約・タスクᴳの完了条件・主種別／副種別と適用経路を確認する。開発タスクᴳは各フェーズᴳ成果物の合意後、非開発タスクᴳは合意した成果物単位で次へ進み、確認境界は CR-073 に従う
- CR-020: リファクタリング時の指摘は依頼範囲内を優先し、重大度と対応要否を添える。依頼範囲外は勝手に修正せず、重要なものだけ報告する
- CR-021: 変数や関数などをリネームする際は使用箇所を確認し、すべて対応する
- CR-022: 変数名や関数名は意味が一意に伝わる名称を用いる。業界・言語で一般的な略語は使用してよい
- CR-023: 情報は役割で分ける: コードに How、テストに What、コミットログに Why、コードコメントには非自明で再検討され得る判断の Why not を記録する
- CR-024: ドキュメント内の空白は半角スペースを用いる（全角スペースは使わない。2026-07-13 指示）
- CR-063: 承認済みタスクᴳ範囲内の `worklog.md` 追記、`resume.md` 最新化、節目 handoverᴳ 作成は事前確認なしで実施し、結果を報告する。その他の確認境界は CR-019 / CR-073 に従う
- CR-070: 最初のファイル編集前に変更対象と変更方針を依頼者へ明示する。対象または方針が実質的に変わった場合は、次の編集前に再度明示する
- CR-071: 日本語ファイルの編集後は、差分または編集後内容を確認し、文字化け・異体字・不可視文字・全角半角の取り違えを点検する
- CR-073: 依頼者との確認は下表の4種に分ける。待機が必要な `合意`・`承認`・`裁定`は、依頼者または依頼者が指定した承認者から明示的な回答を得るまで先へ進めない

| 種別 | 使用場面 | 応答待ち |
| --- | --- | --- |
| 明示 | CR-070 の変更対象・方針、進捗・結果の通知 | 不要。通知後に継続可 |
| 合意 | タスクᴳ定義、フェーズᴳ成果物、非開発タスクᴳの成果物単位の確定 | 必要 |
| 承認 | スコープ拡大、不可逆操作、対外反映、新規依存、状態変更Git、重要な設計変更 | 必要 |
| 裁定 | 監査指摘・判断レンズᴳ・未解決論点の選択 | 必要 |

## AI開発ルール

- CR-025: AI生成コードは、自動化可能な振る舞いに対応するテストと同梱でのみ受け入れる。自動化が不可能または著しく不合理な受入条件ᴳは、理由・再現可能な手順・期待結果・証跡を記録して検証するが、コードテストの代替にはしない（2026-07-11 プロジェクト側 AI開発規約から昇格）
- CR-026: マージ前に人間が全 diff を読む。理解できない箇所は AI に説明させ、腹落ちしなければマージしない（同上）
- CR-027: 新規依存パッケージは AI が理由を提示し、人間の承認後に追加する（同上）
- CR-028: AI が主に書いたコミットはメッセージ末尾に `[ai]` を付ける（品質傾向の事後分析用。同上）
- CR-029: ファイル種別にかかわらず、シークレット・個人情報・マスク前の本番ログ・本番データを AI に渡さない。必要な情報は除去または匿名化し、内容を確認してから渡す
- CR-069: 外部仕様・技術選定には公式文書等の外部一次根拠、影響範囲・内部コード構造には `file:line`・検索結果・実行結果等の内部一次根拠を併記する。確認できないものは `要検証` とし、別モデル監査は一次根拠の代替ではなく補助とする

## 判断レンズᴳ運用（02_Roles）

- CR-030: フェーズᴳ 2（要件）、4（詳細設計）、7（リファクタ後）と、フェーズᴳ 3が最終設計成果物になる品質ゲートでは `reviewer`ᴳ を用いて独立監査する。高リスクな設計・技術選定・優先度・リリース判断では、さらに `~\Projects\02_Roles\` の関連レンズ2〜3個を併用してトレードオフを確認する
- CR-031: レンズの意見が割れたら `~\Projects\02_Roles\_judgment-axis.md`（判断軸）に照らして決める。決まらなければ同ファイル末尾の手続きに従う
- CR-032: レンズは助言であり決定者ではない。最終判断は依頼者または依頼者が指定した承認者が下す
- CR-033: 各レンズの「固有の立場」は業界標準のたたき台。運用で方針が定まったら該当レンズに追記し、末尾の最終確認日を更新する
- CR-034: 独立監査は生成担当と異なるモデル・エージェント・人が行う。利用不能時は依頼者または依頼者指定の監査者へ依頼し、それも不能なら未実施理由を検証待ちキューᴳへ記録して当該品質ゲートを未完了とする。生成担当自身の自己レビューは独立監査に数えない
- CR-035: レンズ一覧・起動条件・出力の型・更新規律の詳細は `~\Projects\02_Roles\README.md` を参照する

## ディレクトリ運用（Projects 直下）

## ドキュメント参照範囲の運用

- CR-037: リポジトリ内ドキュメントは同一リポジトリ内で完結させ、上位階層のファイルへ参照を残さない。上位情報が必要な場合はリポジトリの参照スコープに取り込み、同期関係はプロジェクト階層側で管理する
- CR-038: プロジェクト階層のドキュメント（`PROJECT_RULES.md`・`handover\`・`temp\`等）からリポジトリ内ドキュメントへの参照は可とする（一方向。逆方向は上記により不可）
- CR-039: `~\Projects` 直下の横断ドキュメントと個別プロジェクトディレクトリは、内容・リンク・実在を相互に依存させない。匿名化した構造例は可とし、`NN_`接頭辞ディレクトリ（プロジェクト横断の共有リソース）への参照は例外的に可とする
- CR-064: 数値・基準等の正ᴳは同一参照スコープ内で1文書にのみ置く。自己完結のため別スコープへ複製する場合は各スコープの正ᴳを定め、同期元・基準時点・更新条件を参照可能な上位階層で管理する
- CR-067: 共通用語の定義は `~\Projects\04_Rules_Reference\glossary.md` に集約する。本文で定義済み用語を用語集の定義どおりの意味で使う場合は、文脈上明らかでも全出現箇所に上付き ᴳ（U+1D33）を付け、一般的な意味・用語名そのものの説明・リテラル表記には付けない。ᴳは初出記号ではなく意味標識とし、詳細は glossary.md に従う

## セッションルール

- CR-047: セッションはフェーズᴳの節目で区切る。1 セッションで連続する複数フェーズᴳを進め、次回再開時に現在の会話文脈を保持できない、担当・ツールを切り替える、または CR-058 の節目に達した場合に分割する。CR-068 対象の横断作業は成果物境界で区切り、残件を最終報告へ記載する。1 タスクᴳを必ず 1 セッションに収める必要はない（2026-07-15 改訂。旧: 1 タスクᴳ 1 セッション運用）
- CR-048: プロジェクト作業の分割時は `resume.md` と直近handoverᴳ、CR-068 対象の横断作業の再開時は前タスクᴳの最終報告を精査し、再開に必要な情報を確認してからタスクᴳ定義を再合意する
- CR-049: 開発タスクᴳは 開始手続きᴳ → タスクᴳ定義 → 必要時のタスクᴳ記録初期化・ブランチ名／チャット名提案 → 要件定義 → 基本設計 → 詳細設計 → テスト実装 → 実装 → リファクタ → ドキュメント整備 の順で進める。複数セッションにまたがる場合は途中フェーズᴳから再開し、詳細は `~\Projects\04_Rules_Reference\session-phases.md` に従う
- CR-050: タスクᴳ種別ごとの適用フェーズᴳ・成果物・DoDᴳは `~\Projects\04_Rules_Reference\session-phases.md` の適用表に従う。実装または挙動変更を伴う開発ではフェーズᴳ 2〜7を省略せず、非該当フェーズᴳと理由はタスクᴳ定義へ記録する
- CR-051: 確認は CR-073 の `合意`・`承認`・`裁定`に該当する境界へ集中させ、合意済み工程内の可逆な読取・編集・検証では繰り返さない
- CR-052: 各プロジェクトの `handover\` ディレクトリに、`worklog.md`（重要な作業差分と証跡）、`resume.md`（現在の再開地点）、節目 handoverᴳ（区間要約）を役割分担して持ち、同じ説明を重複記載しない（`00_Template` を除く）
- CR-068: プロジェクトに属さない横断作業では `worklog.md`・`resume.md`・handoverᴳ を作らない。通常の実施内容と未検証事項は最終報告に記載し、`~\Projects\incident-log.md` は失敗・指摘だけに使用する。変更時の run-checks（CR-062）は対象範囲に実施する
- CR-054: 工程完了・重要判断・長時間または不可逆な操作の前後・中断・担当やツールの切替をチェックポイントᴳとし、`worklog.md` へ差分を追記して `resume.md` を最新化する。同じ説明を複数文書へ重複転記しない
- CR-055: `worklog.md` の各エントリには 日時 / 実行ツール・セッション識別 / 実施内容 / 変更ファイル / 決定と理由 / 検証（CR-065 の DoDᴳ 消込形式） / 次の一手 を記す（どのツールが何をしたか追跡するため）
- CR-056: `resume.md` には 更新時刻 / 実行ツール・セッション / 対象ブランチ / 現在タスクᴳとゴール / 直近完了ステップᴳ / 次の一手 / 途中状態 / 未解決・ブロッカー / 検証待ちキューᴳ / 参照先を、次セッションが即再開できる最小量で記す
- CR-057: セッション開始時はプロジェクト側 `PROJECT_RULES.md` → `resume.md`（あれば）→ 必要に応じ `worklog.md` 末尾 の順で読み、再開地点を把握してから着手する。再読込は resume.md のみを既定とし、handoverᴳ・worklogᴳ はブロッカー解消や詳細確認が必要なときだけ引く（分割時の初期コストを抑えるため）
- CR-058: タスクᴳ・機能の完了、次の営業日以降へ持ち越す中断、担当・ツール交代など、現在の会話文脈だけでは再開できない節目で `handover-YYYY-MM-DD[-N].md` を `handover\` ディレクトリに作成し、当該区間のworklogᴳを要約する。短い同一文脈の中断や通常のセッション終了では作成しない
- CR-059: `worklog.md` の再読が困難になった時は当該区間を節目 handoverᴳ へ要約し、`worklog-YYYY-MM[-N].md` へ重複しない名称で退避して本体を軽く保つ。既存退避ファイルは上書きせず、`resume.md` は常に短く保つ
- CR-060: ブランチを用いるプロジェクト開発の開始時に、タスクᴳ内容が分かるブランチ名とチャット名を提示する。読取専用レビュー・単発相談・横断ルール作業では必須としない

- 位置づけ: 状況別ルールリファレンス。Cowork のサンドボックス Bash / ホスト側ツールを併用するセッションで、作業前に読む
- 参照元: `04_Rules_Reference\README.md` の参照トリガー
- 更新: 追記・変更・削除はすべて提案 → 依頼者承認後に反映する。採番は `CROSS_PROJECT_RULES.md` の ID採番ルールに従う

## ルール

- CS-001: マウント先での削除系（`os.remove` / `os.rmdir` / `rm`）は使わず、移動・置換は「内容書換 → `os.rename`（上書きリネーム可）」で行う（2026-07-11: docs 配置換えで削除系のみ EPERM、rename は成功）
- CS-002: ホスト側ツールで書いた直後のファイルがサンドボックス側の読取で途切れて見えることがある。実破損と判断する前にホスト側 Read で実ファイルを確認する（2026-07-11: 書込直後の index.md が 100 バイトに見えたが実ファイルは正常）
- CS-003: ホスト側 Edit が「modified since read」エラーになったら、対象ファイルを Read で再読込してから再実行する（フォルダ接続直後に頻発。2026-07-11: 3回観測、いずれも再読込で解消）
- CS-004: ホスト側ツールの編集でファイルが伸びた直後は、サンドボックス読取が旧サイズで末尾欠落することがある。当該ファイルのサンドボックス検証は行わず、依頼者ローカルでの実行・確認を正ᴳとする。サンドボックス側で内容が必要な場合は新規パスへ書き出して使う（2026-07-14: 同一セッションで4ファイルに発生）
- CS-005: サンドボックスで実行する危険文字スキャンは、対象文字をリテラルではなくエスケープ表記（`\uXXXX`）で記述する。heredoc 転送時に不可視文字リテラルが別の文字へ化け、スキャン自体が誤検出・見逃しを起こすため（2026-07-17: NBSPリテラルが半角スペースへ化け、全行を誤検出する偽陽性5273件が発生）。エスケープ表記でも `bash -c '...python3 - <<EOF...'` 経由だと化ける事例が再々発したため、確実を期すならスキャンスクリプトを `.py` ファイルに `Write` で書き出してから `python3 <path>` で実行する（2026-07-17: heredocでNBSPリテラル化けが3回再発、ファイル化実行で解消。スクリプトは scratchpad の `outputs/` 等へ書き出す）

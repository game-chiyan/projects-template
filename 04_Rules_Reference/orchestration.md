# orchestration.md（Cowork オーケストレーション運用）

- 位置づけ: 状況別ルールリファレンス。サブエージェント委譲の運用ルールと手順・判断基準・雛形を持つ。`CLAUDE.md` の参照トリガー表から読む
- 対象: Cowork（Claude Agent SDK / `Agent` ツール）でサブエージェントを使う作業
- 優先順位: 方針が矛盾する場合は `CLAUDE.md` → `CROSS_PROJECT_RULES.md` に従う
- 更新: 追記・変更・削除はすべて提案 → 依頼者承認後に反映する。採番は `CROSS_PROJECT_RULES.md` の ID採番ルールに従う

## 用語

- オーケストレータ: タスクᴳを分解し、サブエージェントへ委譲・統合する主体。メインセッション（原則として上位ティア）が担う
- サブエージェント: `Agent` ツールで起動する下位エージェント。親の文脈を共有せず、返るのは最終メッセージのみ
- ネスト段数: メインを 0 段とし、サブを 1 段、サブが起動したサブを 2 段と数える

## 運用ルール

- OR-001: タスクᴳは原則すべてサブエージェントへ委譲する（OR-010）。分解要否と委譲先は「工程の独立性」と「主ティアと各工程が要するティアの差」で決め、独立工程は並列（OR-009）、下位で足りる工程は下位ティアへ（OR-008）、難所は上位ティアへ（OR-007）。極小工程のみメイン直接処理を許容する（上位ティア使用量の温存と二重コスト回避を両立するため。2026-07-14 策定 / 2026-07-16 改訂）
- OR-002: オーケストレーションはメインセッションで統括し、サブタスクを適材ティアへ委譲する。メインの上に別オーケストレータを重ねない（二重起動は冗長。メインのティアが不足する難所は上位ティアへ委譲して補う。2026-07-14 策定）
- OR-003: モデルは固有名で指名せず役割ティアで選ぶ: 高速ティア = 探索・抽出・一括読取、均衡ティア = 実装・分析、上位ティア = 難所の推論・最終統合、監査 = 生成と別系統のティア。委譲時は使うティアと役割をプロンプトに明示する（名指しを避け新モデルに追従。2026-07-14 策定）
- OR-004: ティアと現行モデルの対応・能力順・コストは本書「役割ティアと現行モデル対応表」に集約し、新モデル登場時は対応表だけを更新する。能力順・使用量残を実行時に自動取得する手段は無いため、能力順は依頼者に確認する（自動判定不可を人手で補完。2026-07-14 策定）
- OR-005: サブ成果物の一次レビューはメインが担う（OR-010）。重要判断では加えて `~\Projects\02_Roles\` の reviewerᴳ レンズを生成と別系統のモデルで起動し監査ゲートとする（同一モデルの相関した盲点を避け、既存レンズ運用へ統合する。2026-07-14 策定 / 2026-07-16 調整）
- OR-006: サブエージェントのネストは最大 2 段までとし、ファイル書込み・不可逆操作・Git はサブに委ねずメインと依頼者承認に集約する（サブは文脈非共有で結果は最終メッセージのみ返るため、深いネストは精度が落ちる。書込みの集約で「変更は提案→承認」を担保。2026-07-14 策定）
- OR-007: 指定ティアが手に余っても自動昇格はされない。限界を明示し、難所サブタスクの上位ティア委譲・別ティア監査・スコープ縮小を依頼者へ提案する（下位から上位への委譲は可能と確認済み。自己認識は不確実なため人手チェックポイントで補う。2026-07-14 策定）
- OR-008: 主セッションが上位ティア（例: 最高能力 = Fable 5）で開始された場合、そのティアを要しないサブタスク（実装・分析・変換・探索・抽出）は下位ティア（均衡 = Sonnet / 高速 = Haiku）のサブエージェントへ委譲し、上位ティアの使用量を温存する。主には難所推論と最終統合のみ残す（上位ティア主の直列処理による使用量浪費を防ぐため。2026-07-16 指示）
- OR-009: 独立サブタスクは同一ターン（1メッセージ内に複数の `Agent` 呼び出し）でまとめて起動し並列に走らせる。1つ起動→結果待ち→次と別ターンに分けると直列化するため避け、依存があるものだけ前段の返却後に起動する（別ターン逐次起動が直列化の主因。2026-07-16 明文化）
- OR-010: メインセッションは実作業（生成・実装・分析・変換・探索・抽出）を自ら行わず、オーケストレーションと監査に専念する: 分解・委譲・サブ成果物のレビュー・品質不足時の修正指示・最終統合・ファイル書込みの確定（OR-006）。実作業は必ずサブエージェントを起動して委譲し、例外は極小工程（1行編集・単一ファイルの短時間読取など委譲往復の方が高コストな場合）のみメイン直接処理を許容する（上位ティア主の直列処理による使用量浪費を構造的に防ぐため。2026-07-16 指示）
- OR-011: 委譲先ティアが使用上限・エラーで停止した場合、メインは上位ティアで肩代わりせず、状況を明示して待機・スコープ縮小・別ティア委譲を依頼者に確認する（上位ティア温存の目的が崩れるのを防ぐため。2026-07-16 指示）
- OR-012: レビュー→修正の反復は新規起動せず同一サブエージェントを継続（SendMessage で文脈保持）して指示する（文脈非共有による毎回の再転記でメインの入力トークンを浪費しないため。2026-07-16 指示）

## 起動判断フロー

1. タスクᴳを受けたら、実作業は原則サブエージェントへ委譲する前提で工程を洗い出す（OR-010）。メインは分解・委譲・監査・統合・書込み確定に専念する
2. 各工程の必要ティアを見積もり、主ティアが上回る工程は下位ティアへ（OR-008）、難所は上位ティアへ（OR-007）割り当てる
3. 独立工程は同一ターンでまとめて並列起動する（OR-009）
4. 依存がある工程だけ前段の返却を受けてから次段を起動する
5. 極小工程（1行編集・単一ファイルの短時間読取など）に限り、委譲往復が高コストならメインで直接処理してよい

## 役割ティアと現行モデル対応表

- 固有名で指名せず役割ティアで選ぶ。委譲時は使うティアと役割をプロンプト冒頭に明示する
- ティアと現行モデルの対応は下表に集約し、新モデル・新ファミリー登場時は本表だけを更新する

| 役割ティア | 用途 | 現行モデル（2026-07 時点・要保守） | コスト(入/出 MTok) | 速度 |
| --- | --- | --- | --- | --- |
| 最高能力 | 最難関の推論・長時間エージェント | Claude Fable 5 | $10 / $50 | 低速 |
| エージェント既定 | 複雑なエージェント・実装の既定 | Claude Opus 4.8 | $5 / $25 | 中 |
| 均衡 | 実装・分析・変換全般 | Claude Sonnet 5 | $3 / $15 | 速い |
| 高速 | 探索・抽出・一括読取 | Claude Haiku 4.5 | $1 / $5 | 最速 |
| 監査 | 完了物の批判的レビュー | 生成と別系統の上位ティア（例: 生成 Sonnet なら監査 Opus / Fable） | 生成物依存 | - |

- 能力順（一般提供）: Claude Fable 5 > Claude Opus 4.8 > Claude Sonnet 5 > Claude Haiku 4.5（出典: Anthropic Models overview）。最難関は最高能力ティア、複雑なエージェント・実装の既定はエージェント既定ティアを使う
- `Agent` ツールの `model` で選べる別名は現在 opus / sonnet / haiku / fable。新ファミリーはハーネス側が別名を追加して初めて選べるため、登場時に本表と併せて確認する
- 能力順・使用量残を実行時に自動取得する手段はこのセッションに無い（モデル一覧 API は能力情報を返すが知能順・残量は返さず、本セッションからは呼べない）。能力順は依頼者に確認し、残量は保守的スコープで扱う
- 使用量枠はモデル別に配分される。Claude Fable 5 は週次使用制限の最大 50% までとされる（出典: @claudeai 公式ポスト https://x.com/claudeai/status/2074548243971604641 ）。上位ティアを実作業で消費すると枠を早く使い切るため、下位委譲（OR-008・OR-010）で温存する意義がある

## 委譲プロンプトの雛形

サブは文脈を共有しないため、プロンプトだけで完結させる。最低限、次を含める。

- 役割と使うティア（例: 「あなたは均衡ティアの実装担当」）
- 背景・前提（親が持つ情報のうち必要分を明示的に転記する）
- 入力（対象ファイルの絶対パス・データ）
- 成果物と受入基準（何をどこまで。判断が要る場合はその基準）
- 返却フォーマット（親が統合しやすい構造。要点・変更点・未解決を分けて返す指定）
- 禁止事項（ファイル書込み・不可逆操作・Git はしない等、後述のガードレール）

## 並列実行と結果統合

- 独立サブタスクは同一ターン（1メッセージ内の複数 `Agent` 呼び出し）で起動し並列に走らせる（OR-009）。別ターンに分けると直列化する
- 返却は最終メッセージのみのため、各サブに構造化フォーマットで返させ、メインで突き合わせる
- 食い違いや欠落があれば、メインで判断するか、該当サブのみ再委譲する
- レビュー→修正の反復は同一サブエージェントを SendMessage で継続し、文脈を再送しない（OR-012）

## 監査フロー（別モデル）

1. 完了物がまとまったら、`~\Projects\02_Roles\` の reviewerᴳ レンズを別モデルで起動する
2. 監査観点は reviewerᴳ レンズの定義に従い、対象の受入基準・要件との整合を批判的に確認させる
3. 重要判断では reviewerᴳ に加え関連レンズも併用する（個数と運用は `CROSS_PROJECT_RULES.md`「判断レンズ運用」に従う）
4. 監査で指摘が出たら、該当箇所を修正して再監査するか、依頼者に判断を仰ぐ

## 指定ティアが手に余る場合

- 自動昇格はされない。メインセッションのティアが不足しても上位ティアへ勝手に切り替わらず、指定ティアのまま実行を試みる。品質低下・誤り・不完全な結果になり得る（誤りに気づかず自信ありげに答えることもある）
- 補い方: 難所サブタスクを上位ティアのサブエージェントへ委譲する（下位から上位への委譲は sonnet から opus で確認済み）。ただし「手に余る」の自己認識は不確実なため、既知の難タスクは初めから上位ティアで開始するか、別ティア監査ゲートを必須にする
- 兆候: 堂々巡り・浅い回答・整合の崩れ。別ティアの reviewerᴳ 監査で捕捉する

## 難所エスカレーションのプロトコル

- 現行モデルで難しいと判断したら、無理に進めず限界を明示する（何が難しいか・なぜか）
- 選択肢を提示する: 上位モデルへのエスカレーション / 別モデルでの監査・セカンドオピニオン / スコープ縮小
- 依頼者の裁定を得てから進める（`CROSS_PROJECT_RULES.md`「作業ルール」の都度確認に従う）

## ガードレール

- ネストは最大 2 段まで。深い自律ツリーを作らない（文脈非共有と伝言ゲームで精度が落ちるため）
- ファイル書込み・不可逆操作・Git はサブに委ねず、メインと依頼者承認に集約する
- サブには読取・探索・分析・草案作成までを担わせ、確定反映はメインで行う
- 使用量(quota)の実行時取得 API は無い。長時間・広範なツリーは着手前にスコープを見積もり、区切りごとにチェックポイントᴳを置く

## 既知の制約

- 文脈非共有: サブは親の履歴を見ない。必要情報はプロンプトへ明示転記する
- 返却は最終メッセージのみ: 途中経過は親から見えない。構造化返却で補う
- コスト・レイテンシ: 段数と並列数に比例して増える。最小構成を優先する
- 版・残量の非可視: 「最上位」や残量を実行時に確認できない。エイリアスと保守的運用で扱う

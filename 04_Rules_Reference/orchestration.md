# orchestration.md（AIエージェント・オーケストレーション運用）

- 位置づけ: 状況別規範。サブエージェント委譲の運用ルール、判断基準、手順、雛形を持つ。参照トリガーは `04_Rules_Reference\README.md` を正ᴳとする
- 対象: サブエージェント機能を持つAIツールで、並列化・専門分離・独立監査を行う作業
- 優先順位: `CROSS_PROJECT_RULES.md` 冒頭の優先順位に従う
- 更新: 追記・変更・削除はすべて提案 → 依頼者承認後に反映する。採番は `CROSS_PROJECT_RULES.md` の ID採番ルールに従う

## 用語

- オーケストレータ: タスクᴳを分解し、サブエージェントへ委譲・統合する主体。メインセッションが担う
- サブエージェント: 現在のAIツールが提供する委譲機能で起動する下位エージェント。文脈共有・途中出力・継続方法はツールごとに異なるため、必要な入力と返却形式を明示する
- ネスト段数: メインを 0 段とし、サブを 1 段、サブが起動したサブを 2 段と数える

## 運用ルール

- OR-001: サブエージェントは、独立工程を並列化できる場合、異なる専門性へ分離する価値がある場合、または品質ゲートの独立監査に使用する。連続した文脈が品質に直結する作業や、再説明コストが成果を上回る作業はメインが直接処理する
- OR-002: オーケストレーションはメインセッションで統括し、サブタスクを適材ティアへ委譲する。メインの上に別オーケストレータを重ねない（二重起動は冗長。メインのティアが不足する難所は上位ティアへ委譲して補う。2026-07-14 策定）
- OR-003: モデルは固有名で指名せず役割ティアで選ぶ: 高速ティア = 探索・抽出・一括読取、均衡ティア = 実装・分析、上位ティア = 難所の推論・最終統合、監査 = 生成と別系統のティア。委譲時は使うティアと役割をプロンプトに明示する（名指しを避け新モデルに追従。2026-07-14 策定）
- OR-004: 現行モデルとティアの対応・能力・料金は変動情報として実行時の公式情報または利用環境で確認し、本書へ固定値を持たせない
- OR-005: サブ成果物の一次レビューはメインが担う。CR-030 の品質ゲートではreviewerᴳを生成担当と異なるモデル・エージェント・人で起動し、高リスク判断だけ関連レンズを追加する（CR-030 / CR-034）
- OR-006: サブエージェントのネストは最大2段とする。書込みを委ねる場合は変更範囲を独立させ、メインが差分統合と確認境界を管理する。不可逆操作・対外反映・状態変更Gitはサブに委ねず、CR-073 の承認後にメインが行う。読み取り専用Gitは調査に使用してよい（CR-061）
- OR-007: 指定ティアが手に余っても自動昇格はされない。限界を明示し、難所サブタスクの上位ティア委譲・別ティア監査・スコープ縮小を依頼者へ提案する（下位から上位への委譲は可能と確認済み。自己認識は不確実なため人手チェックポイントᴳで補う。2026-07-14 策定）
- OR-008: 自己完結し、必要な文脈を短く渡せ、品質低下が見込まれない工程は下位ティアへ委譲してよい。品質・監査・引継ぎを使用量削減より優先する
- OR-009: ツールが対応する場合、独立サブタスクは同一ターンまたは同一バッチでまとめて起動し並列に走らせる。依存があるものだけ前段の返却後に起動する（別ターン逐次起動が直列化の主因。2026-07-16 明文化）
- OR-010: OR-001の条件でオーケストレーションする場合、メインは分解・委譲・一次レビュー・統合・書込み確定を担う。委譲条件に当てはまらない工程はメインが直接処理してよい
- OR-011: 委譲先が上限・エラー・品質不足で停止した場合は、同一サブの継続、別ティアへの再委譲、メインでの処理を品質基準で選ぶ。スコープ・費用・納期・重要設計が実質的に変わる場合は CR-073 の承認または裁定を得る
- OR-012: レビュー→修正の反復は、ツールに継続機能があれば同一サブエージェントを継続して指示する。継続不能時だけ新規起動し、必要な文脈を最小限に再提示する（2026-07-16 指示）

## 起動判断フロー

1. 工程を洗い出し、独立並列・専門性分離・品質ゲート監査のいずれかに該当するか判定する
2. 該当しない工程はメインが直接処理する。該当する工程は必要な専門性とティアを見積もる
3. 品質低下がなく自己完結する工程は下位ティア、難所は上位ティアへ割り当てる
4. ツールが対応する場合、独立工程は同一ターンまたは同一バッチでまとめて並列起動し、依存工程だけ前段の返却後に起動する
5. CR-030 の品質ゲートでは生成担当と異なるreviewerᴳを起動する

## 役割ティア

| 役割ティア | 用途 |
| --- | --- |
| 高速 | 探索・抽出・定型変換 |
| 均衡 | 実装・分析・文書化 |
| 上位 | 難所の推論・設計・最終統合 |
| 監査 | 生成担当と別系統による批判的レビュー |

- 委譲時は固有モデル名ではなく役割ティアと担当を指定する
- 現行モデルとの対応、能力、料金、使用量制限は実行時に確認する

## 委譲プロンプトの雛形

文脈共有の有無に依存せず、委譲指示だけで再現できるよう最低限、次を含める。

- 役割と使うティア（例: 「あなたは均衡ティアの実装担当」）
- 背景・前提（親が持つ情報のうち必要分を明示的に転記する）
- 入力（対象ファイルの絶対パス・データ）
- 成果物と受入基準（何をどこまで。判断が要る場合はその基準）
- 返却フォーマット（親が統合しやすい構造。要点・変更点・未解決を分けて返す指定）
- 禁止事項と確認境界（許可した書込み範囲、不可逆操作・対外反映・状態変更Gitは行わない等）

## 並列実行と結果統合

- ツールが対応する場合、独立サブタスクは同一ターンまたは同一バッチで起動し並列に走らせる（OR-009）
- 各サブに構造化フォーマットで返させ、メインで根拠・変更範囲・未解決事項を突き合わせる
- 食い違いや欠落があれば、メインで判断するか、該当サブのみ再委譲する
- レビュー→修正の反復は同一サブエージェントの継続機能を優先する（OR-012）

## 監査フロー（別モデル）

1. CR-030 の品質ゲート成果物がまとまったら、reviewerᴳ（`~\Projects\02_Roles\reviewer.md`）を生成担当と異なるモデル・エージェント・人で起動する
2. 独立した監査者を利用できない場合は、依頼者または依頼者指定の監査者へ依頼する。それも不能なら CR-034 に従って未実施を記録し、品質ゲートを未完了とする
3. フェーズᴳ 2では要件漏れと受入条件ᴳ、フェーズᴳ 4では設計整合・影響範囲・非機能要件、フェーズᴳ 7では最終diff・テスト・AC充足・回帰を監査する
4. 高リスク判断では reviewerᴳ に加え関連レンズ2〜3個を併用する
5. 指摘は `02_Roles\README.md` の監査表で記録し、修正・再監査または依頼者裁定まで追跡する

## 指定ティアが手に余る場合

- 自動昇格はされない。メインセッションのティアが不足しても上位ティアへ勝手に切り替わらず、指定ティアのまま実行を試みる。品質低下・誤り・不完全な結果になり得る（誤りに気づかず自信ありげに答えることもある）
- 補い方: 難所サブタスクを上位ティアへ委譲する。ただし「手に余る」の自己認識は不確実なため、既知の難タスクは初めから上位ティアで開始するか、別系統の監査ゲートを必須にする
- 兆候: 堂々巡り・浅い回答・整合の崩れ。別ティアの reviewerᴳ 監査で捕捉する

## 難所エスカレーションのプロトコル

- 現行モデルで難しいと判断したら、無理に進めず限界を明示する（何が難しいか・なぜか）
- 選択肢を提示する: 上位モデルへのエスカレーション / 別モデルでの監査・セカンドオピニオン / スコープ縮小
- スコープ拡大・重要設計変更は依頼者の承認、費用・納期の選択や未解決論点は依頼者の裁定を得てから進める（CR-073）

## ガードレール

- ネストは最大 2 段まで。深い自律ツリーを作らない（文脈非共有と伝言ゲームで精度が落ちるため）
- 書込みを委ねる場合は対象を重複させず、メインが差分を確認して統合する
- 不可逆操作・対外反映・状態変更Gitはサブに委ねず、CR-073 の承認後にメインが行う
- 長時間・広範なツリーは着手前にスコープを見積もり、区切りごとにチェックポイントᴳを置く

## 既知の制約

- 文脈共有の差: 親履歴・ファイル状態・途中出力の共有範囲はツールごとに異なる。必要情報は委譲指示へ明示する
- 継続機能の差: 同一サブの継続ができないツールでは、新規起動時に必要な文脈を再提示する
- コスト・レイテンシ: 段数と並列数に比例して増える。最小構成を優先する
- 版・残量の変動: モデル能力・料金・使用量制限は実行環境と公式情報で確認する

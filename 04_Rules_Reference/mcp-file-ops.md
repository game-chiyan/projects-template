# mcp-file-ops.md（MCP ファイル操作の既知の注意・filesystem サーバ共通）

- 位置づけ: 状況別ルールリファレンス。MCP ファイル操作を使うセッションで、作業前に読む
- 参照元: `CLAUDE.md` の参照トリガー表
- 更新: 追記・変更・削除はすべて提案 → 依頼者承認後に反映する。採番は `CROSS_PROJECT_RULES.md` の ID採番ルールに従う

## ルール

- MF-001: パス指定は二重バックスラッシュでエスケープする
- MF-002: 読み取り系がタイムアウトする場合は `read_multiple_files`（単一ファイルでも1要素配列）で代替する（旧構成の `read_file`・`get_file_info` で恒常的に発生。2026-07-04 まで観測）
- MF-003: 存在確認は list_directory 系で行う（get_file_info のタイムアウト回避）
- MF-004: list_directory は非再帰。サブディレクトリは個別に呼ぶか `directory_tree` / `search_files` を使う
- MF-005: 新しいパスへ書き込む前に `create_directory` を呼ぶ
- MF-006: `write_file` は既存ファイルを警告なしに上書きする。新規作成時は事前に同名ファイルの不在を list_directory 系で確認する
- MF-007: `edit_file` は直前に対象ファイルを再読込し、oldText は実テキストからの転記とする（記憶やレンダリング表示からの転記は照合失敗の元。波ダッシュ「〜」・不等号「<」（全角「＜」と取り違え）等の類似文字は特に注意。2026-07-04 に「〜」の照合ミスで一括編集が不発。2026-07-05 に不等号「<」を全角と取り違えて照合失敗）
- MF-008: `edit_file` の複数編集はアトミック: 1件でも照合失敗すると全件不発でファイルは無傷（2026-07-04 観測）。失敗時はファイルを再読込し、正確な文字列で再実行する
- MF-009: 日本語ファイルを `edit_file` した後は diff を目視確認する（Unicode の稀な破損対策。「冪等性」「趨勢」「圏/圈」「随/隨」等で破損の観測実績）。自分が新規に書き込む日本語にも適用し、異体字に加えて全角/半角の取り違え（全角チルダ「～」・全角数字「１」「３」・全角記号等）も確認する（異体字混入 2026-07-05、全角/半角の自己混入3件 2026-07-05/06 に観測）
- MF-010: `move_file` が「実行成功・応答エラー（ENOENT）」となることがある。エラー時はリトライ前に list_directory で実状態を確認する（2026-07-04 観測）
- MF-011: MCP サーバ構成の変更直後は、①`list_allowed_directories` で許可範囲を確認 → ②軽いファイル操作（list_directory 等）で疎通を確認してから本作業に入る（2026-07-05: サーバ統合直後に許可範囲の応答は正常・ファイル操作のみ全滅の状態を観測）
- MF-012: ファイル操作のタイムアウトが2連続したらリトライを止め、依頼者に MCP サーバ（または AI ツール）の再起動を依頼する（1回4分の待機損失を抑える。2026-07-05: 再起動で回復を確認）
- MF-013: 日本語ファイルの読み取り表示が文字化けして見えても、即座に内容を修正せず `edit_file` の dry-run で実ファイルの内容を確認する（表示上の問題で実ファイルは正常だったケースがある、2026-07-06: `Hoge` の表示が文字化けして見えたが実ファイルは正常で、誤修正しかけた）
- MF-014: filesystem サーバの許可範囲がセッション途中で作業対象外（例: uploads）に変わり、設定修正・再起動でも即時回復しない場合、リトライを続けず Cowork のフォルダ接続（`request_cowork_directory`）で対象フォルダを直接マウントして作業を継続する。MCP側の恒久対応は別途行う（2026-07-15・2026-07-16: 許可範囲が uploads へ変化し、config修正後も当該セッションに反映されなかった）

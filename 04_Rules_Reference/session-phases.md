# session-phases.md（セッションのフェーズ定義）

- 対象: `~\Projects` 配下で AI ツールが 1 セッションを進行する全作業
- 位置づけ: `CROSS_PROJECT_RULES.md` の CR-049 が参照する詳細定義。CR-049 は流れと参照先のみを持ち、各フェーズᴳの中身は本書に一元化する（CR-010 / CR-064）
- 使い方: 開始手続きᴳを済ませてからフェーズᴳ 0 以降を順に進む。各フェーズᴳの「アウトプット」は次フェーズᴳの「インプット」になる。前フェーズᴳのアウトプットが未確定のまま次へ進まない
- 確認規則: 各フェーズᴳの完了条件にある依頼者合意は必須。途中の確認は随時挟む（CR-051）

## 開始手続き（固定・合意不要）

セッション開始時に毎回固定で実行する。依頼者合意は要さない。

1. ルールの確認: 読み順（入口ファイル → `CROSS_PROJECT_RULES.md` → `PROJECT_RULES.md`）でルールを把握する
2. 引継ぎ資料の読込: `resume.md` のみを既定で読む。handover・`worklog.md` はブロッカー解消や詳細確認が必要なときだけ引く（CR-057。分割時の初期コストを抑えるため）

## フェーズとステップの判定基準

- フェーズᴳに切り出す条件（4 つすべてを満たす）: ①独立した引き継ぎ可能な成果物を生む ②固有の完了条件（依頼者合意）を持つ ③そのアウトプットが次フェーズᴳの入力になる ④単独でスキップまたはセッション分割の境界になりうる
- ステップᴳ: 上記を満たさず、フェーズᴳの単一アウトプットへ収束する内部手順

## 共通様式（各フェーズの記述項目）

各フェーズᴳを インプット / ステップᴳ（複数）/ アウトプット / 完了条件 の 4 項目で定義する。フェーズᴳ間の受け渡しを揃えるための固定様式とする。

## 成果物の配置（docs/task-records）

フェーズᴳ 0〜4 の md 成果物はタスクᴳ単位ディレクトリにまとめ、Docusaurus で閲覧・追跡できるようにする。

```
docs/task-records/
  _category_.json            # label:「タスク記録」
  <タスクスラッグ>/            # スラッグはフェーズ 1 のブランチ名と同一
    _category_.json          # label: タスク名
    index.md                 # タスク概要と各成果物へのリンク
    task-definition.md       # sidebar_position 1
    requirements.md          # 2
    basic-design.md          # 3
    detailed-design.md       # 4
```

- 各 md は Front Matter（`title` / `sidebar_position`）を付ける。作業中は `draft: true` を付けて公開ビルドから隠し、フェーズᴳ 8 の確定時に外す（CR-046 / docs-conventions）
- task-recordsᴳ は完了後に書き換えない時点スナップショット。現行の正ᴳは `specs/`・`design/`、決定履歴は `adr/`（不変ルールは ADR と同じ。CR-064）
- ディレクトリは開始手続きᴳ後、フェーズᴳ 1 のブランチ名確定時に作成する

## スキップ規則

- 仕様書・設計資料作成タスクᴳでは、フェーズᴳ 5（テスト作成）・6（実装）・7（リファクタ）は不要（CR-050）。フェーズᴳ 4（詳細設計）は実装を伴う範囲に限り、それ以外は縮小可。フェーズᴳ 8 の docs/ 反映は省かない
- スキップ可否はフェーズᴳ 0 で判定したタスク種別に従う

## フェーズ 0: タスク定義

- インプット: 依頼者の指示 / 開始手続きᴳで読んだ `resume.md` 等の状態
- ステップᴳ:
  1. 理解したタスクを復唱し、認識ズレを潰す
  2. タスク種別を判定する（実装 / 仕様書・設計資料 / ドキュメント / ルール変更）。スキップ規則と DoDᴳ（CR-065）の分岐起点になる
  3. ゴール・スコープ・スコープ外・制約を明確化する
  4. 参照トリガー表に該当する状況があれば該当ファイルを読む
- アウトプット: タスク定義メモ（`task-definition.md`）
- 完了条件: 依頼者がタスク定義に合意する

## フェーズ 1: ブランチ名・チャット名の提案

- インプット: タスク定義メモ
- ステップᴳ:
  1. ブランチ名候補を提示する（CL-001）
  2. チャット名候補を提示する（CR-060。設定は依頼者操作）
  3. ブランチ名の記述部をタスクスラッグᴳとし、`docs/task-records/<タスクスラッグ>/` を作成する
- アウトプット: 確定ブランチ名 / 確定チャット名 / タスク記録ディレクトリ
- 完了条件: 依頼者がブランチ名を承認し、チャット名を設定する

## フェーズ 2: 要件定義

- インプット: タスク定義メモ
- ステップᴳ:
  1. 目的・背景・意図の確認: なぜやるか / 誰のためか / 成功と見なす状態を言語化する 〔確認 1〕
  2. AS-IS 分析: 目的で対象を絞り、現状の構造・データ・運用・課題・制約を整理する。断定の前に実物（ファイル・実行結果・公式ドキュメント）を読む（read-before-assert）。外部依存の前提には根拠を併記する（CR-069）
  3. TO-BE 設計: あるべき姿と AS-IS との差分（ギャップ）を定義する。重要な判断はレンズ＋ reviewerᴳ を併用する（CR-030）
  4. 要件定義: ギャップを埋める機能要件 / 非機能要件 / スコープ外 / 受入条件ᴳを列挙する 〔確認 2〕
- アウトプット: 要件定義メモ（`requirements.md`）
- 完了条件: 依頼者が要件に合意する。加えて外部依存の前提と重要判断の各事実主張に根拠が併記され、未検証は `要検証` として検証待ちキューᴳ（CR-066）へ回してある（CR-069）
- 補足: 受入条件ᴳは ID 付きの表（ID / Given-When / Then / 観点）で書き、合否を客観判定できる形式にする。曖昧語は数値・境界に置き換える。ここで振った AC-IDᴳ はフェーズᴳ 5 のテストへ引き継ぐ

## フェーズ 3: 基本設計（外部設計 / What）

- インプット: 要件定義メモ（機能要件・非機能要件）
- ステップᴳ:
  1. 方式・システム構成の選定: 候補を出しレンズ＋ reviewerᴳ でトレードオフ比較。採用理由と Why not を残す（CR-030 / CR-023）
  2. 機能構成の設計
  3. 外部インターフェース（画面・API・帳票など）の設計
  4. 論理データモデルの設計
  5. 非機能要件への対応方針の設計
- アウトプット: 基本設計書（`basic-design.md`）
- 完了条件: 依頼者が基本設計に合意する。加えて外部依存の前提と重要判断の各事実主張に根拠が併記され、未検証は `要検証` として検証待ちキューᴳ（CR-066）へ回してある（CR-069）
- 補足: 各要件 → 基本設計項目の対応を保ち、対応先のない要件（設計漏れ）が無いことを確認する

## フェーズ 4: 詳細設計（内部設計 / How）

- インプット: 基本設計書
- ステップᴳ:
  1. モジュール・クラス・関数の分割
  2. 内部ロジックの設計
  3. 物理データ設計
  4. エラーハンドリングの設計
  5. 影響範囲（変更対象一覧）の特定 → フェーズ 6（実装）の作業単位になる。変更対象は実コードを検索・確認してから挙げる（read-before-assert。CR-069）
  6. 未決事項を整理し、検証待ちなら `resume.md` のキューへ回す（CR-066）
- アウトプット: 詳細設計書（`detailed-design.md`）
- 完了条件: 依頼者が詳細設計に合意する（実装着手可）。加えて変更対象（影響範囲）の実在確認と外部依存の前提に根拠が併記され、未検証は `要検証` として検証待ちキューᴳ（CR-066）へ回してある（CR-069）
- 補足: 各基本設計項目 → 詳細設計項目の対応を保ち、設計漏れが無いことを確認する

## フェーズ 5: テストの作成（TDD・Red / 仕様書タスクでは不要）

- インプット: 要件定義メモ（受入条件ᴳ・AC-IDᴳ）/ 詳細設計書
- ステップᴳ:
  1. 受入条件ᴳ（AC-IDᴳ）と詳細設計からテスト観点を設計する
  2. AC-IDᴳ → テストの対応表を作り、全 AC にテストを割り当てる（受入条件カバレッジᴳ 100%）
  3. 失敗するテストを実装し、期待通り失敗すること（Red）を確認する（AI 生成コードはテスト同梱で受け入れ。CR-025）
- アウトプット: テストコード（この時点では失敗）/ AC 対応表
- 完了条件: 全 AC にテストが存在し、テストが期待通り失敗する（Red）。実行不可なら検証待ちキューᴳへ（CR-066）

## フェーズ 6: 実装（TDD・Green / 仕様書タスクでは不要）

- インプット: テストコード（失敗中）/ 詳細設計書（変更対象一覧）
- ステップᴳ:
  1. 変更対象と変更方針を依頼者へ明示する（CL-002 / AG-001）
  2. テストを通す最小実装を書く（Green）。AI が主に書いたコミットは末尾に `[ai]` を付ける（CR-028）
  3. 新規依存は理由を提示し承認後に追加する（CR-027）
- アウトプット: 実装コード（全テストがグリーン）
- 完了条件: 全テストがグリーン、構文確認・lint を通す。実行不可なら検証待ちキューᴳへ（CR-065 / CR-066）

## フェーズ 7: リファクタ（TDD・Refactor / 仕様書タスクでは不要）

- インプット: 実装コード / テストコード（グリーン）
- ステップᴳ:
  1. 気になった点をすべて指摘する（CR-020）
  2. リネーム時は使用箇所を確認し、すべて対応する（CR-021）
  3. 再テストで挙動不変を確認する
- アウトプット: リファクタ済みコード
- 完了条件: テストがグリーンである

## フェーズ 8: ドキュメント整備

- インプット: 各フェーズの全アウトプット
- ステップᴳ:
  1. 仕様書化: 設計書の確定分を `docs/specs/`・`docs/design/` へ反映し、追記・変更箇所を明示する（現行の正ᴳの更新。設計書自体は正ᴳにしない。CR-064 / CR-037）
  2. 重要な決定があれば `docs/adr/` に ADR を追加する
  3. task-records/<タスクスラッグ>/ の `draft` を外し、時点スナップショットとして確定する
  4. `worklog.md` へ追記し `resume.md` を最新化する（CR-054 / CR-055 / CR-056）
  5. 節目なら handoverᴳ を作成する（CR-058）
  6. `03_Checks` を実行し違反を修正または報告する（CR-062）
- アウトプット: 更新済み docs/（specs・design・adr）/ 確定した task-recordsᴳ / handoverᴳ（節目時）
- 完了条件: docs/ 仕様書への反映と整合、文字点検・参照整合を満たす（CR-065）

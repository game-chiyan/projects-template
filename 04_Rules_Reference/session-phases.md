# session-phases.md（タスクᴳ進行と開発フェーズᴳ定義）

- 対象: `~\Projects` 配下で行う全タスクᴳ
- 位置づけ: CR-049 / CR-050 / CR-065 が参照する規範手順。タスクᴳ種別、適用経路、開発フェーズᴳ、成果物、DoDᴳの正ᴳとする
- 使い方: 全タスクᴳでフェーズᴳ 0を行い、適用表で以降の経路を決める。実装・挙動変更を伴う開発はフェーズᴳ 2〜7を省略せず、前フェーズᴳの成果物が未合意のまま次へ進まない
- 確認規則: フェーズᴳ成果物は依頼者または依頼者指定の承認者との合意後に確定する。操作単位の確認は CR-019 / CR-051 / CR-073 に従う

## 開始手続きᴳ（固定・合意不要）

セッション開始時に毎回固定で実行する。

1. 入口ファイル → `CROSS_PROJECT_RULES.md` の順で読む
2. 特定プロジェクトが対象の場合だけ `PROJECT_RULES.md` → `resume.md`（あれば）を読む
3. handoverᴳ・`worklog.md` はブロッカー解消や詳細確認が必要な場合だけ読む（CR-057）
4. `04_Rules_Reference\README.md` の参照トリガーに該当する文書を読む

## フェーズᴳとステップᴳの判定基準

- フェーズᴳに切り出す条件（4つすべて）: ①独立した引継ぎ可能な成果物を生む ②固有の完了条件を持つ ③成果物が次フェーズᴳの入力になる ④単独でスキップまたはセッション分割の境界になり得る
- ステップᴳ: 上記を満たさず、フェーズᴳの単一成果物へ収束する内部手順
- 各フェーズᴳは インプット / ステップᴳ / アウトプット / 完了条件 の4項目で記述する

## タスクᴳ種別ごとの適用経路・成果物・DoDᴳ

- フェーズᴳ 0で主種別を1つ、副種別を0個以上選ぶ。主種別の経路を基準に各種別の適用工程を和集合し、番号付きフェーズᴳは昇順で進め、成果物とDoDᴳを合算する
- 実装・挙動変更が主種別または副種別に含まれる場合は、フェーズᴳ 2〜7を省略しない
- 調査が後工程の前提になる場合はフェーズᴳ 0直後、レビューが後工程の成果物を対象にする場合は対象成果物の作成後かつフェーズᴳ 8の前に実施する。複合時の実行順をタスクᴳ定義へ記録する
- 仕様書・設計資料では、新規または変更する最上流から必要なフェーズᴳ 2〜4を適用する。承認済み上位成果物を再利用する場合だけ前段を省略でき、参照先・版／基準時点・今回も有効と判断した根拠をタスクᴳ定義へ記録する
- 非該当フェーズᴳと理由、採用した経路、DoDᴳはタスクᴳ定義へ記録する

| タスクᴳ種別 | 標準経路 | 必須成果物 | DoDᴳ |
| --- | --- | --- | --- |
| 実装・挙動変更 | 0 → 1（記録初期化、名称は必要時）→ 2 → 3 → 4 → 5 → 6 → 7 → 8 | 合意済みタスクᴳ定義、要件、基本設計、詳細設計、検証対応表、テスト、実装、更新文書 | 構文確認、利用可能なlint、定義済み自動テスト、定義した手動検証、全TC-IDᴳの証跡、CR-025、独立監査 |
| 仕様書・設計資料 | 0 → 1（task-recordsᴳを使う場合）→ 選択基準に該当する2〜4 → 8 | タスクᴳ定義、対象となる仕様／設計成果物、必要な対応表・監査記録 | 上位要件と正ᴳへの整合、対応漏れなし、数値・基準の一元化、依頼者合意 |
| ドキュメント | 0 → 成果物作成・変更 → 8 | タスクᴳ定義、更新文書 | 文字点検、リンク・パス整合、差分目視、依頼内容との一致 |
| ルール変更 | 0 → 成果物作成・変更 → 8 | タスクᴳ定義、更新ルール、必要な索引・入口・検査の更新 | ID・参照整合、文書責務の整合、`03_Checks`、差分目視 |
| 調査・レビュー | 0 → 調査／レビュー実施 → 完了報告 | タスクᴳ定義、根拠付き結論、指摘一覧または調査結果 | 対象範囲、一次根拠、結論、未確認範囲、重大度、対応要否を明示 |

### 調査／レビュー実施単位

- インプット: 合意済みタスクᴳ定義 / 複合タスクᴳでは対象となる前段成果物
- ステップᴳ: 対象範囲を固定する → 一次根拠を収集する → 根拠と推論を分けて分析する → 結論・重大度・対応要否・未確認範囲を整理する
- アウトプット: 根拠付き調査結果または指摘一覧。変更を伴う複合タスクᴳでは後続フェーズᴳの入力、読取専用では完了報告とする
- 完了条件: タスクᴳの全TC-IDᴳを直接証跡または指摘結果へ対応付け、依頼者が成果物に合意する

### 非開発成果物の作成・変更実施単位

- 対象: フェーズᴳ 2〜7を適用しないドキュメント・ルール変更。仕様書・設計資料は適用するフェーズᴳ 2〜4で作成する
- インプット: 合意済みタスクᴳ定義 / 必要な調査結果 / 既存の正ᴳ
- ステップᴳ: 正ᴳと依存する参照先を確認する → 合意済み範囲の成果物を作成・変更する → 索引・入口・検査等の依存箇所を同期する → 差分を目視する
- アウトプット: 更新した文書・ルールと、各TC-IDᴳに対応する検証方法・証跡
- 完了条件: 合意済みスコープの成果物が作成され、全TC-IDᴳの検証先が定まり、依頼者が成果物に合意する

## 成果物の配置（docs/task-records）

リポジトリ内の開発タスクᴳでは、フェーズᴳ 0〜5 のmd成果物をタスクᴳ単位ディレクトリへまとめる。横断ルール整備、読取専用レビュー、またはプロジェクト規則で不要とされた作業では作成しない。

```text
docs/task-records/
  _category_.json
  <タスクスラッグ>/
    _category_.json
    index.md
    task-definition.md       # sidebar_position 1
    requirements.md          # 2
    basic-design.md          # 3
    detailed-design.md       # 4
    verification-matrix.md   # 5
```

- 各mdは Front Matter（`title` / `sidebar_position`）を付ける。作業中は `draft: true` とし、フェーズᴳ 8で確定時に外す
- task-recordsᴳは完了後に書き換えない時点スナップショット。現行の正ᴳは `specs/`・`design/`、決定履歴は `adr/` に置く（CR-064）
- フェーズᴳ 0の合意内容は会話内で先に成立させ、フェーズᴳ 1でタスクスラッグᴳを確定してから `task-definition.md` へ保存する

## 検証の共通定義

- 定義済み自動テスト: ①AC-IDᴳに対応する自動テスト ②変更影響に基づく回帰テスト ③プロジェクトまたはCIが標準指定するテスト
- リポジトリ全体の標準テストより狭い範囲だけを実行する場合は、対象範囲・除外範囲・理由を検証対応表へ記録する
- 手動検証は、自動化が不可能または著しく不合理な受入条件ᴳだけに用いる。理由、再現可能な手順、期待結果、実施結果、証跡の所在を記録する
- 手動検証は CR-025 が要求するコードテストの代替にしない
- lintが未導入なら設定・標準コマンドが存在しない根拠を添えて `非該当`、手動対象がなければ `対象なし` と記録する。実施対象があるのに環境制約で実行できない場合だけ検証待ちキューᴳへ記録する
- 全TC-IDᴳを1つ以上のAC-IDᴳまたは直接検証へ対応付け、最終状態と証跡を下記の正規の消込先へ記録する

### TC-IDᴳの最終消込先

| 作業形態 | 消込先 |
| --- | --- |
| task-recordsᴳを作成する | `verification-matrix.md` |
| プロジェクト作業でtask-recordsᴳを作成しない | `worklog.md` |
| CR-068 対象の横断作業 | 最終報告 |

どの消込先でも `TC-IDᴳ / 対応AC-IDᴳまたは直接検証 / 最終状態 / 証跡` を記録し、全行が合格になるまでタスクᴳを完了しない。

## フェーズᴳ 0: タスクᴳ定義

- インプット: 依頼者の指示 / 開始手続きᴳで把握した状態
- ステップᴳ:
  1. 理解した依頼を復唱し、認識差を解消する
  2. 主種別を1つ、副種別を0個以上選ぶ
  3. ゴール・スコープ・スコープ外・制約を明確化する
  4. タスクᴳの完了条件へTC-IDᴳ（TC-NN）を振り、検証方法または必要な証跡を定める
  5. 適用経路、非該当フェーズᴳと理由、DoDᴳを適用表から選ぶ
- アウトプット: 会話内で合意されたタスクᴳ定義。リポジトリ内開発ではフェーズᴳ 1で `task-definition.md` へ保存する
- 完了条件: 依頼者がタスクᴳ定義に合意する

## フェーズᴳ 1: タスクᴳ記録初期化・必要時の名称提案

- インプット: 合意済みタスクᴳ定義
- ステップᴳ:
  1. ブランチを使う場合はブランチ名候補を提示する（CR-060）
  2. チャット名を管理する場合は候補を提示する（CR-060。設定は依頼者操作）
  3. ブランチ名の記述部またはタスクᴳ名からタスクスラッグᴳを定める
  4. リポジトリ内開発では `docs/task-records/<タスクスラッグ>/` を作成し、フェーズᴳ 0の合意内容を `task-definition.md` へ保存する
- アウトプット: 必要時の確定ブランチ名・チャット名 / タスクスラッグᴳ / 必要時のtask-recordsᴳ初期一式
- 完了条件: 使用する名称について依頼者と合意し、必要な記録先へ合意済みタスクᴳ定義が保存されている。記録・名称が不要な場合は理由を記録してスキップする

## フェーズᴳ 2: 要件定義

- インプット: 合意済みタスクᴳ定義
- ステップᴳ:
  1. 目的・背景・意図: なぜ行うか / 誰のためか / 成功と見なす状態を言語化する
  2. AS-IS分析: 現状の構造・データ・運用・課題・制約を整理し、断定前に実物を確認する（CR-069）
  3. TO-BE設計: あるべき姿とAS-ISとの差分を定義する。高リスク判断は関連する判断レンズᴳを併用する
  4. 機能要件 / 非機能要件 / スコープ外 / 受入条件ᴳを列挙する
  5. 各AC-IDᴳを上位のTC-IDᴳへ対応付け、全TC-IDᴳを1つ以上のAC-IDᴳまたは直接検証へ割り当て、自動／手動の検証方式を仮決定する
  6. reviewerᴳで要件漏れ・受入条件ᴳの客観性・根拠を独立監査する
- アウトプット: 要件定義メモ（`requirements.md`）
- 完了条件: 依頼者が要件に合意し、全TC-IDᴳの検証先が定まり、reviewerᴳの指摘が解消済みまたは依頼者裁定済みである。重要な事実主張へ根拠を併記し、未検証事項は検証待ちキューᴳへ記録する
- 補足: 受入条件ᴳは AC-IDᴳ / 上位TC-IDᴳ / Given-When / Then / 観点 / 検証方式で記述し、曖昧語を数値・境界へ置き換える

## フェーズᴳ 3: 基本設計（外部設計 / What）

- インプット: 合意済み要件定義メモ
- ステップᴳ:
  1. 方式・システム構成をトレードオフ比較し、採用理由とWhy notを残す
  2. 機能構成を設計する
  3. 外部インターフェース（画面・API・帳票等）を設計する
  4. 論理データモデルを設計する
  5. 非機能要件への対応方針を設計する
  6. フェーズᴳ 3が今回の最終設計成果物になる場合は、reviewerᴳで要件整合・方式妥当性・非機能要件を独立監査する
- アウトプット: 基本設計書（`basic-design.md`）
- 完了条件: 依頼者が基本設計に合意し、各要件に対応先があり、重要な事実主張へ根拠が併記されている。最終設計成果物になる場合はreviewerᴳの指摘が解消済みまたは依頼者裁定済みであり、未検証事項は検証待ちキューᴳへ記録する

## フェーズᴳ 4: 詳細設計（内部設計 / How）

- インプット: 合意済み基本設計書
- ステップᴳ:
  1. モジュール・クラス・関数を分割する
  2. 内部ロジックを設計する
  3. 物理データを設計する
  4. エラーハンドリングを設計する
  5. 実コードを確認して変更対象一覧を特定する（CR-069）
  6. 未決事項と検証待ちを整理する
  7. reviewerᴳで要件・基本設計との整合、影響範囲、非機能要件を独立監査する
- アウトプット: 詳細設計書（`detailed-design.md`）
- 完了条件: 依頼者が詳細設計に合意し、reviewerᴳの指摘が解消済みまたは依頼者裁定済みである。変更対象の実在確認と重要な事実主張の根拠が記録されている

## フェーズᴳ 5: テスト実装（TDD・Red）

- インプット: 要件定義メモ（受入条件ᴳ・AC-IDᴳ）/ 詳細設計書
- ステップᴳ:
  1. 全AC-IDᴳを自動テストまたは手動検証へ割り当て、全TC-IDᴳの対応先を含む検証対応表を作る
  2. 自動テスト対象では失敗するテストを実装し、期待どおり失敗すること（Red）を確認する
  3. 手動検証対象では自動化しない理由、再現手順、期待結果、証跡の保存先を定義する。Redは非該当と記録する
  4. 定義済み自動テストの範囲と、標準テストから除外する範囲・理由を記録する
- アウトプット: テストコード / 検証対応表（`verification-matrix.md`）
- 完了条件: 受入条件カバレッジᴳが100%で、自動テスト対象は期待どおりRed、手動検証対象は再現可能な手順と期待結果が定義済みであり、依頼者がテスト・検証設計に合意する

## フェーズᴳ 6: 実装（TDD・Green）

- インプット: Red確認済みテストコード / 詳細設計書 / 検証対応表
- ステップᴳ:
  1. CR-070の明示が未実施、または対象・方針が変わった場合は、編集前に変更対象と変更方針を明示する
  2. テストを通す最小実装を書く（Green）
  3. 新規依存は理由を提示し、承認後に追加する（CR-027）
- アウトプット: 実装コード
- 完了条件: 定義済み自動テストがグリーンで、構文確認と利用可能なlintを通し、依頼者が実装成果物に合意する。実施対象があるのに実行不能な項目は検証待ちキューᴳへ記録する

## フェーズᴳ 7: リファクタ（TDD・Refactor）

- インプット: 実装コード / テストコード / 検証対応表
- ステップᴳ:
  1. 依頼範囲内の改善点を重大度と対応要否付きで整理する
  2. リネーム時は使用箇所を確認し、すべて対応する
  3. 定義済み自動テストと定義した手動検証を実施し、挙動不変と受入条件ᴳ充足を確認する
  4. reviewerᴳで最終diff・テスト結果・受入条件ᴳの充足・回帰リスクを独立監査する
  5. 検証対応表へ全TC-IDᴳの最終状態・証跡・監査指摘の対応状況を記録する
- アウトプット: リファクタ済みコード / 最終化した検証対応表
- 完了条件: 定義済み自動テストと定義した手動検証が完了し、reviewerᴳの必須指摘が解消済みまたは依頼者裁定済みで、依頼者がリファクタ後成果物に合意する

## フェーズᴳ 8: ドキュメント整備・完了記録

- インプット: 適用経路で作成した全成果物
- ステップᴳ:
  1. 仕様・設計変更は `docs/specs/`・`docs/design/` 等の現行の正ᴳへ反映する
  2. 重要な決定があれば `docs/adr/` へADRを追加する
  3. リポジトリ内開発ではtask-recordsᴳの `draft` を外し、時点スナップショットとして確定する
  4. プロジェクト作業では `worklog.md` と `resume.md` を更新し、節目に該当する場合だけhandoverᴳを作成する
  5. 横断作業では最終報告へ実施内容・DoDᴳ・未検証事項を記録する
  6. 本書の「TC-IDᴳの最終消込先」に従い、全TC-IDᴳの最終状態と証跡を消し込む
  7. 変更範囲へ `03_Checks` を実行し、違反を修正または報告する
- アウトプット: 更新済みの正ᴳ / 確定したtask-recordsᴳ / 必要な継続記録または最終報告
- 完了条件: 全TC-IDᴳがAC-IDᴳまたは直接検証へ対応して合格し、適用表のDoDᴳをすべて消し込み、必須検証の残件がなく、文字・参照・正ᴳの整合を満たした成果物に依頼者が合意する
